Error Handling

Production code fails gracefully — beginners don't handle errors; seniors do.

Primary Source for this lesson Read this after the lesson: javascript.info — Error Handling, "try...catch". The section on custom errors and rethrowing is especially important for real applications.

1. The Error Object

When something goes wrong in JavaScript, an Error object is thrown. You can create one explicitly with new Error("message") or it can be thrown automatically by the engine.

Every Error object has at minimum three properties:

  • message — the human-readable description you pass in
  • name — the type of error (e.g. "TypeError", "RangeError")
  • stack — a multi-line string showing the call stack at the moment the error was created
const err = new Error("Something went wrong");

console.log(err.message); // "Something went wrong"
console.log(err.name);    // "Error"
console.log(err.stack);
// Error: Something went wrong
//     at <anonymous>:1:13
//     at ...

// Throwing an error stops execution and unwinds the call stack
function divide(a, b) {
  if (b === 0) throw new Error("Cannot divide by zero");
  return a / b;
}

divide(10, 2);  // 5
divide(10, 0);  // throws Error: Cannot divide by zero
The stack property is invaluable for debugging In production, always log err.stack, not just err.message. The stack tells you exactly where in your code the error originated, including all the function calls that led there.

3. try / catch / finally

try wraps the risky code. catch handles errors if any are thrown. finally runs no matter what — whether an error was thrown or not, whether there's a return in try or catch.

function parseJSON(str) {
  try {
    const data = JSON.parse(str);
    console.log("Parsed successfully");
    return data;
  } catch (err) {
    console.log("Failed to parse:", err.message);
    return null;
  } finally {
    console.log("parseJSON finished"); // always runs
  }
}

parseJSON('{"name":"Karan"}');
// Parsed successfully
// parseJSON finished

parseJSON("invalid json");
// Failed to parse: Unexpected token i...
// parseJSON finished

finally always runs — even after return

function test() {
  try {
    return "from try";
  } finally {
    console.log("finally ran!"); // this ALWAYS executes
  }
}

test();
// finally ran!
// "from try"
finally is for cleanup Use finally for cleanup tasks that must happen regardless of success or failure: closing database connections, stopping loading spinners, releasing resources. If you only use try/catch, a thrown error prevents cleanup code from running.

4. Rethrowing Errors

A catch block should only handle errors it knows how to fix. If an unexpected error type arrives, rethrow it so it can propagate to a higher-level handler. Swallowing all errors silently is one of the most harmful patterns in production code.

function processInput(data) {
  try {
    if (typeof data !== "string") {
      throw new TypeError("data must be a string");
    }
    return data.trim().toUpperCase();
  } catch (err) {
    // Handle only what we expect — TypeError from our own validation
    if (err instanceof TypeError) {
      console.warn("Type warning:", err.message);
      return ""; // graceful fallback
    }
    // Something unexpected — rethrow so it doesn't disappear
    throw err;
  }
}

processInput("  hello  "); // "HELLO"
processInput(42);          // "" (TypeError caught and handled)
Never do this try { ... } catch (err) { } — an empty catch block silently buries errors. You will never know what went wrong. At minimum, log the error in every catch block.

5. Custom Error Classes

In real applications you'll have errors that carry more context than a plain message — a field name, an HTTP status code, an error code. Create custom error classes by extending Error.

class ValidationError extends Error {
  constructor(message, field) {
    super(message);         // sets this.message
    this.name = "ValidationError"; // overrides "Error"
    this.field = field;     // extra context
  }
}

class NotFoundError extends Error {
  constructor(resource) {
    super(`${resource} not found`);
    this.name = "NotFoundError";
    this.statusCode = 404;
  }
}

function validateAge(age) {
  if (typeof age !== "number") {
    throw new ValidationError("Age must be a number", "age");
  }
  if (age < 0 || age > 150) {
    throw new ValidationError("Age out of valid range", "age");
  }
}

try {
  validateAge("old");
} catch (err) {
  if (err instanceof ValidationError) {
    console.log(`Validation failed on field "${err.field}": ${err.message}`);
    // Validation failed on field "age": Age must be a number
  } else {
    throw err; // unexpected — rethrow
  }
}
Always set this.name in custom errors If you don't set this.name, all custom errors will log as "Error", which makes debugging much harder. Setting the name makes err.toString() and logs show "ValidationError: ..." instead of just "Error: ...".

6. Error Handling in Async Code

Asynchronous errors cannot be caught with a synchronous try/catch around a callback. But with Promises and async/await, errors are fully catchable.

// PROMISES — use .catch()
fetch("https://api.example.com/data")
  .then(res => res.json())
  .then(data => console.log(data))
  .catch(err => console.error("Fetch failed:", err.message));

// ASYNC/AWAIT — use try/catch
async function loadUser(id) {
  try {
    const res = await fetch(`/api/users/${id}`);
    if (!res.ok) {
      throw new Error(`HTTP error: ${res.status}`);
    }
    const user = await res.json();
    return user;
  } catch (err) {
    // Catches: network errors, JSON parse errors, AND our thrown error
    console.error("loadUser failed:", err.message);
    return null;
  }
}
await turns rejected promises into thrown errors When you await a rejected promise, JavaScript converts the rejection into a regular thrown error. That's why try/catch works perfectly with async/await — rejections and thrown errors are treated identically.

Unhandled Promise Rejections

// If a promise rejection is never caught, the browser/Node emits a warning
// In Node 15+ it actually crashes the process

// Listen for unhandled rejections globally (use in monitoring/logging layers)
window.addEventListener("unhandledrejection", (event) => {
  console.error("Unhandled rejection:", event.reason);
  // Send to your error monitoring service (Sentry, Datadog, etc.)
});

7. Global Error Handling

In production, you need to catch errors that slip through all your try/catch blocks. Two global hooks let you do this:

// Synchronous errors that weren't caught
window.onerror = function(message, source, line, col, error) {
  console.error("Uncaught error:", message, "at", source, line);
  // return true to suppress the default browser error display
  return true;
};

// Unhandled promise rejections
window.addEventListener("unhandledrejection", (event) => {
  console.error("Unhandled promise rejection:", event.reason);
  event.preventDefault(); // suppress default console output
});
