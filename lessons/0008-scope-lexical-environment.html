Scope & Lexical Environment

Understanding scope is the key to understanding where variables live and die.

📖 Primary Source for this lesson Read this after the lesson: javascript.info — Lexical Environment. This section of the closures chapter is the single best explanation of how JS variable lookup really works under the hood.

1. What is Scope?

Scope is the set of rules that determines where a variable can be accessed in your code. Think of it as the variable's visibility boundary — inside the boundary you can see it; outside you cannot.

When JavaScript needs to resolve a variable name, it follows the scope chain: it looks in the current scope first, then walks outward to each enclosing scope, stopping when it finds the variable or reaches the global scope (at which point it throws a ReferenceError).

Mental model Think of scopes like nested rooms with one-way mirrors. An inner room can look out and see the outer room. The outer room cannot look in and see the inner room.

2. Global Scope

Any variable declared outside of any function or block lives in the global scope. In browsers it's attached to the window object; in Node.js it's global.

// These are global variables
var appName = "MyApp";
let version = "1.0";

function showVersion() {
  console.log(version); // ✅ accessible — found in global scope
}

// In browsers: var variables become window properties
console.log(window.appName); // "MyApp"
console.log(window.version);  // undefined — let/const are NOT on window
Polluting global scope Every global variable can be accidentally overwritten by any script on the page (including third-party libraries). Keep as few globals as possible. Wrap your code in modules or IIFEs to isolate it.

Strict Mode Prevents Accidental Globals

"use strict";

function oops() {
  undeclaredVar = 42; // ❌ ReferenceError in strict mode
                        // Without "use strict" this would silently create a global!
}
oops();

3. Function Scope

var is function-scoped. A variable declared with var inside a function is visible everywhere in that function — and only that function.

function makeSecret() {
  var secret = "I'm hidden";
  console.log(secret); // ✅ "I'm hidden"
}

makeSecret();
console.log(secret); // ❌ ReferenceError: secret is not defined

// var is accessible anywhere inside the SAME function, even before its declaration (hoisting)
function weirdVar() {
  console.log(x);  // undefined (hoisted)
  var x = 5;
  console.log(x);  // 5
}

4. Block Scope

let and const are block-scoped. A block is any pair of curly braces { } — including if, for, while, and bare { }.

if (true) {
  let blockVar = "I'm block-scoped";
  const alsoBlock = 42;
  console.log(blockVar); // ✅ "I'm block-scoped"
}

console.log(blockVar);   // ❌ ReferenceError: blockVar is not defined
var leaks out of blocks — this is a bug factory var is only scoped to functions, not blocks. It escapes if statements, for loops, and all other blocks. This is a famous source of bugs. 
// The classic var-in-loop leak
for (var i = 0; i < 3; i++) {
  // i is var — it belongs to the enclosing function/global scope
}
console.log(i); // 3 — leaked out of the for block!

// With let — correct behavior
for (let j = 0; j < 3; j++) {
  // j is let — scoped to the for block
}
console.log(j); // ❌ ReferenceError: j is not defined

5. Lexical Scope (Static Scope)

Lexical scope means that the scope of a variable is determined by where the code is written, not by where the function is called from. Inner functions can access variables from any outer scope they are nested within.

const global = "I'm global";

function outer() {
  const outerVar = "I'm outer";

  function inner() {
    const innerVar = "I'm inner";

    // inner can see all three — scope chain: inner → outer → global
    console.log(innerVar);  // "I'm inner"
    console.log(outerVar);  // "I'm outer"
    console.log(global);    // "I'm global"
  }

  inner();
  // outer cannot see innerVar
  console.log(innerVar);   // ❌ ReferenceError
}

outer();
Lexical vs. Dynamic scope JavaScript uses lexical (static) scope — determined at write time. Some languages use dynamic scope (determined at call time). Understanding this distinction is why you can predict what a function will "see" just by reading the source code, regardless of how or where it's called.

6. Variable Shadowing

When an inner scope declares a variable with the same name as an outer variable, the inner variable shadows the outer one. Inside that inner scope, the outer variable is completely inaccessible by that name.

const name = "Global Karan";

function greet() {
  const name = "Local Karan"; // shadows the outer name
  console.log(name);          // "Local Karan" — inner one wins
}

greet();
console.log(name);            // "Global Karan" — outer scope unchanged

// Shadowing in a for loop — a very common and intentional pattern
const i = "I'm the outer i";
for (let i = 0; i < 3; i++) {
  console.log(i); // 0, 1, 2 — the loop's i, not the outer const i
}
console.log(i); // "I'm the outer i"
Shadowing can be a bug or a feature Shadowing in a for loop is fine and intentional. Shadowing a critical variable like user or token in a deeply nested function is a source of subtle bugs. Use your linter (no-shadow ESLint rule) to catch unintentional shadowing.

7. The Lexical Environment (How JS Really Works)

Every time a function is called (or a block is entered), JavaScript creates a Lexical Environment — a structure that stores two things:

  1. An Environment Record: a mapping of all local variable names to their current values
  2. A reference to the outer Lexical Environment: a pointer to the enclosing scope

When JS needs to resolve a variable, it looks in the current Lexical Environment's record. If not found, it follows the outer reference, then that outer's outer, and so on — all the way to the global Lexical Environment. This chain of references is the scope chain.

// Visualizing Lexical Environments as nested boxes:
//
// [ GLOBAL LE: x=10, outer() ]
//   |
//   └── [ outer() LE: y=20, inner() | outer ref → GLOBAL ]
//         |
//         └── [ inner() LE: z=30 | outer ref → outer() LE ]

const x = 10;

function outer() {
  const y = 20;

  function inner() {
    const z = 30;
    // Looking up x: not in inner's LE → check outer's LE → not there → check global LE → found!
    console.log(x + y + z); // 60
  }
  inner();
}
outer();
Why this matters — preview of closures When inner() returns, normally its Lexical Environment would be garbage collected. But what if a function created inside outer() is returned and stored somewhere? It keeps a reference to outer()'s Lexical Environment alive, even after outer() has finished executing. That is a closure — covered in full in the next lesson.
