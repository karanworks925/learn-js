Fetch API & Browser Storage

Network requests and data persistence — the two foundations of every real web app.

Primary Source for this lesson Read this after the lesson: javascript.info — Fetch and javascript.info — localStorage. The fetch article covers error handling patterns in depth — worth reading twice.

1. The fetch() API

fetch() is the modern browser API for making HTTP requests. It replaced the older XMLHttpRequest (XHR) and is Promise-based, making it easy to use with async/await.

There is a subtlety that catches almost every developer the first time: fetch has two steps.

// Step 1: fetch() gives you a Response object — the headers arrived
// Step 2: .json() reads the body stream and parses it — the body arrived
async function getUser(id) {
  const response = await fetch(`https://api.example.com/users/${id}`);
  const user = await response.json(); // this is also async
  console.log(user);
}

// What the Response object contains before you call .json()
response.status   // 200, 404, 500, etc.
response.ok       // true if status is 200–299
response.headers  // Headers object
response.url      // final URL (after redirects)
Why two awaits? HTTP responses come in as a stream. The first await resolves when the headers arrive — at that point the body may still be downloading. Calling .json() (or .text(), .blob()) reads the full stream and resolves when the body is complete.

2. The Critical Gotcha: fetch Does NOT Reject on HTTP Errors

This is the most important thing to know about fetch, and the most common source of silent bugs.

SituationDoes fetch() reject?response.ok
Server returns 200 OKNo (resolves)true
Server returns 404 Not FoundNo (resolves!)false
Server returns 500 Server ErrorNo (resolves!)false
No internet / DNS failureYes (rejects)N/A
CORS blockedYes (rejects)N/A

The correct pattern — always check response.ok:

async function fetchUser(id) {
  try {
    const res = await fetch(`/api/users/${id}`);

    // MUST check this — fetch won't throw for 404/500
    if (!res.ok) {
      throw new Error(`HTTP error: ${res.status}`);
    }

    const data = await res.json();
    return data;
  } catch (err) {
    // catches BOTH network failures AND our thrown HTTP error
    console.error('Fetch failed:', err.message);
    throw err; // re-throw so the caller can handle it
  }
}
Silent failure trap If you skip the response.ok check and the server returns a 404, your code will happily call response.json() and parse the error body (often HTML). You'll get confusing data or a JSON parse error — not the clear error message you expected.

3. POST Requests — Sending Data to a Server

Pass a second argument to fetch() with the method, headers, and body.

async function createUser(userData) {
  const res = await fetch('/api/users', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${token}`
    },
    body: JSON.stringify(userData) // object → JSON string
  });

  if (!res.ok) throw new Error(`${res.status}`);
  return res.json();
}

// Usage
createUser({ name: 'Karan', email: 'karan@example.com' })
  .then(user => console.log('Created:', user))
  .catch(err => console.error(err));
Don't forget Content-Type If you omit the Content-Type: application/json header, many servers won't know how to parse the body and will return a 400 Bad Request. Always set it when sending JSON.

4. JSON.stringify and JSON.parse

JSON (JavaScript Object Notation) is the standard data format for web APIs. These two functions are the bridge between JS objects and JSON strings.

const user = { name: 'Karan', age: 22, active: true };

// Object → JSON string (for sending over the network or storing)
const jsonString = JSON.stringify(user);
// '{"name":"Karan","age":22,"active":true}'

// JSON string → Object (for reading from the network or storage)
const parsed = JSON.parse(jsonString);
// { name: 'Karan', age: 22, active: true }

// Pretty-printing for debugging (3rd arg = indentation)
console.log(JSON.stringify(user, null, 2));

// What stringify CANNOT handle:
JSON.stringify({ fn: () => {}, sym: Symbol() });
// → '{}' — functions and symbols are silently dropped

JSON.stringify(undefined); // → undefined (not a string)

// Circular references throw:
const obj = {};
obj.self = obj;
JSON.stringify(obj); // ❌ TypeError: circular structure

5. localStorage — Persistent Client-Side Storage

localStorage stores data in the browser that persists indefinitely — even after the tab closes and the browser restarts. The key API:

// Storing a string
localStorage.setItem('theme', 'dark');

// Reading it back
const theme = localStorage.getItem('theme'); // 'dark'
const missing = localStorage.getItem('nope');  // null (not undefined)

// Removing one item
localStorage.removeItem('theme');

// Nuking everything
localStorage.clear();

// CRITICAL: localStorage only stores strings
// Store an object correctly:
const user = { name: 'Karan', score: 42 };
localStorage.setItem('user', JSON.stringify(user));

// Read it back correctly:
const savedUser = JSON.parse(localStorage.getItem('user'));

// Safe read with fallback:
function getFromStorage(key, fallback = null) {
  try {
    const item = localStorage.getItem(key);
    return item ? JSON.parse(item) : fallback;
  } catch {
    return fallback;
  }
}
The string-only trap localStorage.setItem('user', { name: 'Karan' }) stores the string "[object Object]" — the result of calling .toString() on the object. Always wrap objects in JSON.stringify() before storing.

7. Cookies — When the Server Needs to Know

Cookies are the oldest client-side storage mechanism. Their defining characteristic: they are automatically sent to the server with every HTTP request to the same domain. This makes them the right tool for authentication tokens and server-session tracking.

// Reading cookies (awkward string format)
console.log(document.cookie);
// "theme=dark; userId=42; sessionToken=abc123"

// Setting a cookie from JS
document.cookie = "theme=dark; path=/; max-age=86400";
// max-age in seconds: 86400 = 1 day

// Cookies have attributes that localStorage doesn't:
// expires / max-age — when it expires
// path — which URL paths the cookie is sent to
// domain — which domain it belongs to
// Secure — only sent over HTTPS
// HttpOnly — cannot be read by JS (server sets this for security)
// SameSite — CSRF protection (Strict / Lax / None)

// You CANNOT set HttpOnly from JavaScript — servers set it via Set-Cookie headers
// This is intentional: HttpOnly cookies protect auth tokens from XSS
Storage TypeSent to Server?LifetimeCapacityBest For
localStorageNoForever~5MBUser preferences, cached data
sessionStorageNoTab session~5MBTemporary form state, wizard steps
CookiesYes (auto)Configurable~4KBAuth tokens, server sessions
Decision rule If the server needs to see the data — use cookies. If it's purely client-side — use localStorage or sessionStorage. For auth: let the server set HttpOnly cookies so JavaScript can't steal the token even if there's an XSS vulnerability.
