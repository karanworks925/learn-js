1. What Are Design Patterns?

Design patterns are reusable solutions to commonly occurring software design problems. They're not code you copy-paste — they're conceptual templates: proven approaches you adapt to your situation.

They matter in your career for two reasons:

  1. Shared vocabulary. When a senior engineer says "that's a Strategy pattern" or "use an EventEmitter here", you need to know what they mean — and be able to say it yourself.
  2. Proven structure. When you recognize a pattern, you avoid reinventing the wheel and avoid the specific mistakes that pattern was designed to prevent.

CreationalHow objects are createdSingleton, Factory, Builder

2. Module Pattern

The Module pattern creates private state using closures and exposes only a public API. Before ES modules existed, this was the way to encapsulate code in JavaScript.

// Classic Module Pattern (IIFE + closure)
const Counter = (function() {
  // Private — not accessible from outside
  let count = 0;

  function validate(n) {
    return typeof n === 'number';
  }

  // Public API — returned object
  return {
    increment() { count++; },
    decrement() { count = Math.max(0, count - 1); },
    add(n) { if (validate(n)) count += n; },
    getCount() { return count; }
  };
})(); // IIFE — immediately invoked

Counter.increment();
Counter.increment();
console.log(Counter.getCount()); // 2
console.log(Counter.count);       // undefined — private!

// Modern equivalent: ES module (use this in new code)
// counter.js
let count = 0;
export const increment = () => count++;
export const getCount = () => count;

3. Singleton Pattern

The Singleton ensures that only one instance of something is ever created. Subsequent calls return the same instance.

// Singleton using a closure
const AppConfig = (function() {
  let instance; // stored in closure — private

  function createInstance() {
    return {
      apiUrl: 'https://api.example.com',
      timeout: 5000,
      debug: false
    };
  }

  return {
    getInstance() {
      if (!instance) {
        instance = createInstance(); // create once
      }
      return instance; // return cached forever after
    }
  };
})();

const cfg1 = AppConfig.getInstance();
const cfg2 = AppConfig.getInstance();
console.log(cfg1 === cfg2); // true — same object
Singletons and testing don't mix well Because a singleton holds global state, tests that run in sequence can bleed state into each other. It's fine for app-wide config, but use dependency injection instead of singletons for services that need to be mockable in tests.

4. Observer / Pub-Sub Pattern

One of the most important patterns in front-end development. It decouples producers of events from consumers.

Observer: the subject (publisher) directly calls methods on its observers. They know each other.
Pub-Sub: events travel through a broker/event bus. Publisher and subscriber never reference each other directly — fully decoupled.

// Simple EventEmitter (pub-sub broker)
class EventEmitter {
  constructor() {
    this.events = {}; // { eventName: [handler, handler, ...] }
  }

  on(event, handler) {
    if (!this.events[event]) this.events[event] = [];
    this.events[event].push(handler);
    return this; // chainable
  }

  off(event, handler) {
    if (!this.events[event]) return;
    this.events[event] = this.events[event]
      .filter(h => h !== handler);
  }

  emit(event, data) {
    (this.events[event] || []).forEach(h => h(data));
  }
}

// Usage
const bus = new EventEmitter();

// Subscriber A — doesn't know about B
bus.on('userLoggedIn', ({ username }) => {
  console.log(`Welcome, ${username}!`);
});

// Subscriber B — also doesn't know about A
bus.on('userLoggedIn', ({ username }) => {
  logToAnalytics('login', username);
});

// Publisher — doesn't know who's listening
bus.emit('userLoggedIn', { username: 'karan' });
// Both subscribers fire
You already use this pattern DOM's addEventListener is pub-sub. Node.js's EventEmitter is pub-sub. Redux's store subscription is observer. React's Context updates are observer. This is one of the most widely used patterns in all of software.

5. Factory Pattern

A Factory is a function (or method) that creates and returns objects without the caller needing to know the class or use new. It centralizes object creation logic and can return different types based on input.

// Factory function — no 'new' required by the caller
function createUser(type, name) {
  const base = {
    name,
    createdAt: new Date().toISOString(),
    greet() { return `Hi, I'm ${this.name}`; }
  };

  if (type === 'admin') {
    return { ...base, role: 'admin', permissions: ['read', 'write', 'delete'] };
  }
  if (type === 'editor') {
    return { ...base, role: 'editor', permissions: ['read', 'write'] };
  }
  return { ...base, role: 'viewer', permissions: ['read'] };
}

const admin = createUser('admin', 'Karan');
const viewer = createUser('viewer', 'Raj');

console.log(admin.permissions); // ['read', 'write', 'delete']
console.log(admin.greet());    // "Hi, I'm Karan"
Factory vs Constructor Constructor functions (or classes with new) always return the same type of object. Factories can return different shapes based on input, can return objects that have already been created (caching), and don't require the new keyword — which means no errors if the caller forgets it.

6. Strategy Pattern

The Strategy pattern defines a family of interchangeable algorithms (strategies) and lets you swap between them at runtime — without changing the code that uses them.

It eliminates long if/else or switch chains that select behavior, replacing them with objects you swap in.

// Without Strategy — messy and hard to extend
function processPayment(amount, method) {
  if (method === 'stripe') { /* stripe logic */ }
  else if (method === 'paypal') { /* paypal logic */ }
  else if (method === 'crypto') { /* crypto logic */ }
  // Adding a 4th method means editing this function
}

// With Strategy — each algorithm is its own object
const paymentStrategies = {
  stripe: {
    pay(amount) { console.log(`Stripe: charged $${amount}`); }
  },
  paypal: {
    pay(amount) { console.log(`PayPal: sent $${amount}`); }
  },
  crypto: {
    pay(amount) { console.log(`Crypto: transferred $${amount} in BTC`); }
  }
};

function processPayment(amount, method) {
  const strategy = paymentStrategies[method];
  if (!strategy) throw new Error(`Unknown payment method: ${method}`);
  strategy.pay(amount);
}

processPayment(99, 'stripe');  // "Stripe: charged $99"
processPayment(99, 'paypal');  // "PayPal: sent $99"
// Adding a 4th: just add to the object — no if/else changes needed

7. Anti-Patterns to Avoid

An anti-pattern is a common response to a recurring problem that seems reasonable but makes things worse. Knowing them is as important as knowing the patterns.

Anti-patternWhat it looks likeThe problem
God ObjectOne class/module that handles everything: auth, UI, data, networkingViolates single responsibility; impossible to test or maintain
Callback HellDeeply nested callbacks: a(b(c(d(e()))))Unreadable, error-prone; use async/await instead
Global Namespace PollutionAttaching everything to window: window.myApp = ...Name collisions, hard to debug, tight coupling
Premature OptimizationSpending hours optimizing code that isn't a bottleneck"Premature optimization is the root of all evil" — Knuth. Profile first.
Spaghetti CodeNo structure, logic scattered everywhere, no separation of concernsUnmaintainable; every change breaks something else
The God Object is the most common senior-dev red flag In code reviews, one of the first things a senior engineer looks for is a class or file that's doing too much. If your App.js imports 40 things and has 1000 lines, that's a God Object. Break it up by concern — each module should have one clear job.
