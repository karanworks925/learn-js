Callbacks & Promises

From callback hell to the Promise chain — the evolution of async JavaScript.

Primary Source for this lesson Read after the lesson: javascript.info — Promise Basics. Then continue with Promise Chaining and Promise API.

1. The Callback Pattern

Before Promises, all asynchronous JavaScript relied on callbacks — functions passed as arguments to be called later, when the async work finishes. This pattern still appears everywhere in older code and in Node.js built-in APIs.

// Simple callback — wait 1 second, then run the function
setTimeout(function() {
  console.log("One second has passed.");
}, 1000);

// Callbacks continue running after the async work is done
console.log("This runs immediately.");

Node.js Error-First Convention

Node.js standardized a callback signature where the first argument is always an error (or null if there was no error), and the second is the result. This convention is called error-first callbacks or Node callbacks.

const fs = require('fs');

// Signature: callback(error, result)
fs.readFile('./data.txt', 'utf8', function(err, data) {
  if (err) {
    console.error("Failed to read file:", err);
    return; // always return early on error
  }
  console.log("File contents:", data);
});
Why Error-First? The error-first convention forces you to handle errors — you cannot accidentally skip them. The error is always argument zero, so you always see it. If there is no error, it is null, and you proceed. This became the universal standard in the Node.js ecosystem.

2. Callback Hell — The Pyramid of Doom

The real problem with callbacks becomes clear when you need to perform several async operations in sequence — where each step depends on the result of the previous one. The callbacks nest inside each other, creating a shape that developers called the "pyramid of doom."

// Three sequential async operations — already painful to read
getUser(userId, function(err, user) {
  if (err) { return handleError(err); }

  getOrders(user.id, function(err, orders) {
    if (err) { return handleError(err); }

    getInvoice(orders[0].id, function(err, invoice) {
      if (err) { return handleError(err); }

      // Finally we have what we need — buried 4 levels deep
      console.log(invoice);
    });
  });
});

Notice the problems with this code:

  • Each error must be handled separately — there is no way to write one centralized error handler
  • Hard to read — the logic flows diagonally instead of top-to-bottom
  • Hard to add steps — inserting a new async operation in the middle requires restructuring everything
  • Hard to reuse — the inner callbacks are tightly coupled to their position in the nesting
Callback Hell in the Real World Real callback hell in production codebases could have 8-10 levels of nesting, each with its own error handling. It was genuinely dangerous — a missing return on an error branch would cause the callback to continue executing, creating hard-to-find bugs.

3. Promises — The Concept

A Promise is an object that represents the eventual result of an asynchronous operation. Think of it like a receipt: when you order coffee, you get a receipt (a Promise) immediately. You don't have the coffee yet, but you have a guarantee that it will either arrive (fulfilled) or you'll be told it's unavailable (rejected).

The Three States

StateMeaningCan it change?
pendingThe async operation has not finished yetYes — can become fulfilled or rejected
fulfilledThe operation completed successfully; has a valueNo — final
rejectedThe operation failed; has a reason (error)No — final
States are Final Once a Promise is settled (fulfilled or rejected), it will never change state again. A fulfilled Promise cannot later become rejected. This predictability is one of the key advantages over callbacks.

4. Creating Promises

You create a Promise by calling new Promise() with an executor function. The executor receives two arguments: resolve (call with the success value) and reject (call with the error reason).

const myPromise = new Promise((resolve, reject) => {
  // The executor runs immediately and synchronously
  const success = true;

  if (success) {
    resolve("Operation succeeded!"); // fulfills the Promise
  } else {
    reject(new Error("Something went wrong")); // rejects the Promise
  }
});

Practical Example — Wrapping setTimeout in a Promise

This is the canonical example for understanding how to "promisify" a callback-based API:

function delay(ms) {
  return new Promise(resolve => {
    setTimeout(resolve, ms); // resolve is called after ms milliseconds
  });
}

// Usage — much cleaner than nested callbacks
delay(1000).then(() => {
  console.log("One second has passed!");
});
The Executor Runs Synchronously The function you pass to new Promise() runs immediately and synchronously. Only the resolution (calling resolve or reject) notifies the .then() handlers, which run as microtasks.

5. Consuming Promises

A Promise exposes three methods for handling its eventual result:

const promise = fetchUserData(userId);

// .then() — handles fulfillment
promise.then(user => {
  console.log("Got user:", user.name);
});

// .catch() — handles rejection (preferred over second arg of .then)
promise.catch(err => {
  console.error("Failed:", err.message);
});

// .finally() — runs always, regardless of outcome
// Does not receive the value or reason
promise.finally(() => {
  hideLoadingSpinner(); // cleanup that should always happen
});

Chaining All Three Together

fetchUserData(userId)
  .then(user => {
    console.log("User:", user.name);
    return user; // pass to the next .then()
  })
  .catch(err => {
    console.error("Error:", err.message);
  })
  .finally(() => {
    hideSpinner();
  });
.then(null, onError) vs .catch(onError) .catch(fn) is essentially shorthand for .then(null, fn), but there is a subtle difference: .catch() placed after a .then() will also catch errors thrown inside that .then() handler. Using .then(onFulfilled, onRejected) as two arguments does NOT catch errors thrown by onFulfilled.

6. Promise Chaining

Every .then() call returns a new Promise. This is what makes chaining work — and it is the most important mechanic to understand about Promises.

  • If you return a plain value from .then(), the next Promise resolves with that value
  • If you return a Promise from .then(), the chain waits for that Promise to settle before continuing
  • If you throw inside .then(), the chain jumps to the nearest .catch()
// Callback hell version
getUser(id, (err, user) => {
  getOrders(user.id, (err, orders) => {
    getInvoice(orders[0].id, (err, invoice) => {
      console.log(invoice);
    });
  });
});

// Promise chain version — flat, readable, one error handler
getUser(id)
  .then(user => getOrders(user.id))     // returns a Promise — chain waits
  .then(orders => getInvoice(orders[0].id)) // returns a Promise — chain waits
  .then(invoice => console.log(invoice))   // receives the final value
  .catch(err => console.error(err));      // catches ANY error in the chain
The Classic Mistake — Broken Chain Forgetting to return inside a .then() breaks the chain. The next .then() receives undefined instead of the value. Always return from .then() handlers that perform async work.

7. Promise Combinators

JavaScript provides four static methods for coordinating multiple Promises. Choosing the right one is an interview question and a practical daily decision.

MethodSettles whenOn rejectionUse case
Promise.all() ALL fulfill Immediately rejects with first reason Need all results; any failure is fatal
Promise.allSettled() ALL settle (either way) Never rejects — always fulfills Need all results; failures are ok
Promise.race() FIRST to settle (either way) Rejects if first to settle is a rejection Timeout patterns; first response wins
Promise.any() FIRST to FULFILL Rejects only if ALL reject (AggregateError) First success wins; try multiple sources 
const p1 = fetchUser(1);
const p2 = fetchUser(2);
const p3 = fetchUser(3);

// Promise.all — run in parallel, get all results
Promise.all([p1, p2, p3])
  .then(([user1, user2, user3]) => {
    console.log("All three users:", user1, user2, user3);
  })
  .catch(err => console.error("At least one failed:", err));

// Promise.allSettled — never fails, inspect each result
Promise.allSettled([p1, p2, p3])
  .then(results => {
    results.forEach(result => {
      if (result.status === "fulfilled") {
        console.log("Success:", result.value);
      } else {
        console.log("Failed:", result.reason);
      }
    });
  });
// Promise.race — timeout pattern
const timeout = new Promise((_, reject) =>
  setTimeout(() => reject(new Error("Request timed out")), 5000)
);

Promise.race([fetchData(), timeout])
  .then(data => console.log(data))
  .catch(err => console.error(err)); // "Request timed out" if fetch is slow
