Events & Event Delegation

Events are the heartbeat of interactive UIs — delegation is what makes them scale.

Primary Source for this lesson Read this after the lesson to deepen understanding: javascript.info — Events. Pay special attention to the bubbling diagram — it clicks everything else into place.

1. addEventListener — The Right Way to Attach Events

There are three ways to respond to user events in the browser. Only one of them is correct for production code.

MethodExampleProblem
HTML attributeonclick="doThing()"Mixes behavior into HTML. Impossible to remove.
DOM propertyel.onclick = fnOnly one listener per event — the second overwrites the first.
addEventListenerel.addEventListener('click', fn)None. This is the standard.

addEventListener accepts a third argument — either a boolean (capture phase) or an options object:

const btn = document.querySelector('#myBtn');

// Basic usage
btn.addEventListener('click', function(event) {
  console.log('clicked!', event);
});

// Options object — the modern way
btn.addEventListener('click', handleClick, {
  once: true,     // auto-removes after first call
  capture: false, // use bubbling phase (default)
  passive: true   // tells browser: I won't call preventDefault()
                   // (scroll performance hint)
});

// Multiple listeners on the SAME event — both fire
btn.addEventListener('click', logClick);
btn.addEventListener('click', trackAnalytics);
Performance hint Mark scroll and touch listeners as passive: true whenever you don't need preventDefault(). The browser can then optimize scrolling without waiting to see if you'll interrupt it.

2. The Event Object

When an event fires, the browser passes an Event object to your handler. It contains everything you need to know about what happened.

Property / MethodWhat it tells you
event.targetThe element that triggered the event (could be a deep child)
event.currentTargetThe element the listener is attached to (always stable)
event.typeThe event name: "click", "keydown", etc.
event.preventDefault()Stop the browser's default action (form submit, link nav)
event.stopPropagation()Stop the event from bubbling further up the DOM
event.keyWhich key was pressed (keyboard events only)
event.clientX / clientYMouse coordinates relative to viewport
 
document.querySelector('a').addEventListener('click', function(event) {
  event.preventDefault(); // don't navigate to the href
  console.log('Link clicked but navigation blocked!');
});

document.querySelector('form').addEventListener('submit', function(e) {
  e.preventDefault(); // don't do a page reload
  handleFormData(e.target); // e.target is the <form> element
});
target vs currentTarget — the classic confusion Imagine a <button> with a <span> inside it. If the user clicks the span, event.target is the <span> — the thing actually clicked. event.currentTarget is the <button> where your listener lives. This distinction is critical for event delegation.

3. Event Bubbling & Capturing

When you click an element deep in the DOM, the event doesn't just fire on that element — it travels. There are three phases:

  1. Capture phase — the event travels DOWN from window to the target
  2. Target phase — the event fires on the target element
  3. Bubble phase — the event travels UP from the target back to window

By default, addEventListener hooks into the bubble phase (the most useful one). Pass true or { capture: true } to hook into the capture phase instead.

// HTML: <div id="outer"><div id="inner">Click me</div></div>

document.getElementById('inner').addEventListener('click', () =>
  console.log('inner')
);
document.getElementById('outer').addEventListener('click', () =>
  console.log('outer')
);

// Click "inner" → logs: "inner", then "outer"
// The event bubbles up from inner to outer

// To stop it bubbling:
document.getElementById('inner').addEventListener('click', (e) => {
  e.stopPropagation(); // outer's listener will NOT fire
  console.log('inner only');
});
Mental model Think of bubbling like dropping a stone in water. The stone hits the bottom (target), then the ripple spreads outward (upward through ancestors). Capturing is the reverse — the ripple arriving before it hits the target.

4. Event Delegation — One Listener to Rule Them All

Instead of attaching a listener to every child element, attach one listener to the parent and use event.target to figure out which child was clicked. This works because events bubble.

Without delegation (bad at scale):

// 100 todo items = 100 listeners. And new items get none.
document.querySelectorAll('.todo-item').forEach(item => {
  item.addEventListener('click', handleItemClick);
});

With delegation (correct approach):

// ONE listener on the parent handles ALL current AND future items
const list = document.querySelector('#todo-list');

list.addEventListener('click', function(event) {
  // event.target is whatever was actually clicked
  // Use .closest() to handle clicks on child elements safely
  const item = event.target.closest('.todo-item');
  if (!item) return; // clicked outside any item — ignore

  console.log('Todo clicked:', item.dataset.id);
  item.classList.toggle('done');
});

// Adding a new item dynamically — the listener ALREADY covers it
function addTodo(text) {
  const li = document.createElement('li');
  li.className = 'todo-item';
  li.dataset.id = Date.now();
  li.textContent = text;
  list.appendChild(li);
  // No need to add a new listener — delegation handles it
}
Why .closest() and not just event.target? If your .todo-item contains a <span>, clicking the span makes event.target the span, not the item. .closest('.todo-item') walks UP from whatever was clicked until it finds the matching ancestor — giving you the right element every time.

Benefits of event delegation:

  • Works automatically for dynamically added elements — no re-binding needed
  • Fewer event listeners in memory — better performance for large lists
  • Single place to manage behavior for a whole group of elements

Mouseclick, dblclick, mousedown, mouseup, mouseover, mouseout, mousemovemouseover/mouseout bubble; use mouseenter/mouseleave for no-bubble versions

// DOMContentLoaded — your typical "run when DOM is ready" hook
document.addEventListener('DOMContentLoaded', () => {
  initApp();
});

// Keyboard: detect Enter key
document.addEventListener('keydown', (e) => {
  if (e.key === 'Enter') console.log('Enter pressed');
  if (e.key === 'Escape') closeModal();
});

// input fires on every keystroke — great for live search
document.querySelector('#search').addEventListener('input', (e) => {
  filterResults(e.target.value);
});

6. Removing Event Listeners

To remove a listener, you must pass the exact same function reference that was used to add it. This is the most common mistake with event listener cleanup.

// BAD — this does NOT work
btn.addEventListener('click', () => console.log('hi'));
btn.removeEventListener('click', () => console.log('hi'));
// ^ new arrow function = new reference. removeEventListener can't find it.

// GOOD — store the reference
function handleClick() {
  console.log('hi');
}
btn.addEventListener('click', handleClick);
// Later, when cleanup is needed:
btn.removeEventListener('click', handleClick); // ✅ works

// EASIEST — use { once: true } to auto-remove after first call
btn.addEventListener('click', handleOnce, { once: true });
Memory leak warning Forgetting to remove event listeners from elements that get removed from the DOM can cause memory leaks. This is especially relevant in single-page apps where components mount and unmount. Always clean up in your component's unmount/destroy lifecycle.

7. Custom Events

You can create and dispatch your own events — a clean way to communicate between different parts of your application without tight coupling.

// Create a custom event with attached data
const event = new CustomEvent('userLoggedIn', {
  detail: { username: 'karan', role: 'admin' },
  bubbles: true,    // event will bubble up the DOM
  cancelable: true  // can be preventDefault()'d
});

// Dispatch it from any element
document.dispatchEvent(event);

// Listen for it anywhere
document.addEventListener('userLoggedIn', (e) => {
  console.log('Welcome,', e.detail.username); // 'Welcome, karan'
  updateNavbar(e.detail);
});

// Real-world pattern: a component fires an event,
// another component listens — no direct coupling needed
When to use Custom Events Custom events are great for decoupled component communication — a dropdown that announces "item selected" without knowing who cares. But in larger apps, a dedicated state manager (like Redux or Zustand) is often cleaner. Custom events shine in vanilla JS or Web Component architectures.
