Iterators, Generators & Symbols

Custom iteration, lazy evaluation, and JavaScript's hidden metaprogramming hooks.

Primary Sources for this lesson Read these after the lesson: javascript.info — Generators and javascript.info — Symbol. These are dense topics — reading both articles after the lesson will solidify what you learn here significantly.

1. Symbols — Guaranteed Unique Keys

A Symbol is a primitive value introduced in ES6. Its defining property: every Symbol is unique, even if created with the same description.

// Creating symbols
const s1 = Symbol('id');
const s2 = Symbol('id');

console.log(s1 === s2); // false — ALWAYS. Symbols are unique.
console.log(typeof s1); // 'symbol'
console.log(s1.toString()); // 'Symbol(id)' — the description

// Use case: unique property key that can't accidentally collide
const ID = Symbol('id');
const user = {
  name: 'Karan',
  [ID]: 42   // computed property key using a symbol
};

console.log(user[ID]);   // 42 — accessed only via the symbol
console.log(user.id);    // undefined — string 'id' ≠ Symbol('id')

// Symbol properties are hidden from normal enumeration
console.log(Object.keys(user));  // ['name'] — no ID symbol
JSON.stringify(user);              // '{"name":"Karan"}' — symbol dropped

Well-Known Symbols — JavaScript's Metaprogramming Hooks

JavaScript itself uses special built-in symbols called well-known symbols to define how objects behave with language features. You override them to customize that behavior.

Well-Known SymbolWhat it customizes
Symbol.iteratorMakes an object work with for...of, spread, destructuring
Symbol.toPrimitiveControls type coercion (how an object converts to number/string)
Symbol.hasInstanceControls what instanceof returns for your class
Symbol.toStringTagChanges the tag shown in Object.prototype.toString
Why a symbol and not a string? If the iterable protocol used a string like "iterator", any object property named "iterator" would accidentally become an iterable. Using a symbol guarantees that these hooks never collide with regular object properties — they exist in a separate namespace.

2. The Iterable Protocol

An object is iterable if it implements the iterable protocol: it must have a method at [Symbol.iterator] that returns an iterator.

The following are iterable by default because JavaScript has built-in [Symbol.iterator] implementations on them:

  • Arrays
  • Strings
  • Maps and Sets
  • Generator objects
  • NodeList (DOM collections)
  • Function arguments object

This is why for...of, spread ([...x]), and destructuring (const [a, b] = x) all work on arrays and strings but not on plain objects — plain objects have no [Symbol.iterator] method.

// These all rely on the iterable protocol:
const arr = [1, 2, 3];

for (const n of arr) { console.log(n); } // uses Symbol.iterator

const copy = [...arr];   // spread uses Symbol.iterator
const [a, b] = arr;     // destructuring uses Symbol.iterator

// Plain object is NOT iterable — this throws:
const obj = { x: 1, y: 2 };
for (const v of obj) {} // ❌ TypeError: obj is not iterable

// Manually accessing the iterator:
const iter = arr[Symbol.iterator]();
console.log(typeof iter.next); // 'function'

3. The Iterator Protocol

An iterator is any object with a next() method. Each call to next() returns an object with exactly two properties:

  • value — the current value
  • done — true if iteration is finished, false otherwise
// Manually using an array's iterator
const arr = ['a', 'b', 'c'];
const iter = arr[Symbol.iterator]();

console.log(iter.next()); // { value: 'a', done: false }
console.log(iter.next()); // { value: 'b', done: false }
console.log(iter.next()); // { value: 'c', done: false }
console.log(iter.next()); // { value: undefined, done: true }

// for...of is just sugar for this loop:
let result;
while (!(result = iter.next()).done) {
  console.log(result.value);
}

4. Making a Custom Iterable

You can make any object iterable by adding a [Symbol.iterator] method. That method must return an object with a next() method that follows the iterator protocol.

// A range object that iterates from 'from' to 'to' inclusive
const range = {
  from: 1,
  to: 5,

  [Symbol.iterator]() {
    // 'this' is the range object
    let current = this.from;
    const last = this.to;

    // Return the iterator object
    return {
      next() {
        if (current <= last) {
          return { value: current++, done: false };
        }
        return { value: undefined, done: true };
      }
    };
  }
};

// Now it works with ALL iteration features
for (const n of range) {
  console.log(n); // 1, 2, 3, 4, 5
}

console.log([...range]); // [1, 2, 3, 4, 5]
const [first, second] = range; // first=1, second=2
The analogy Think of an iterable as a book — it knows there are pages to read. The iterator is a bookmark — it tracks your current position. You can create multiple bookmarks (iterators) from one book (iterable), each at different places.

5. Generator Functions

Generators are a special kind of function that can pause execution and yield values one at a time. They are the most elegant way to write custom iterators.

Three things that make generators different:

  1. Declared with function* (note the asterisk)
  2. Use yield to produce a value and pause
  3. Calling a generator function does NOT run the body — it returns a generator object that is both iterable AND an iterator
// A simple generator
function* threeValues() {
  console.log('start');
  yield 1;           // pauses here, returns { value: 1, done: false }
  console.log('middle');
  yield 2;           // pauses here, returns { value: 2, done: false }
  console.log('end');
  yield 3;           // pauses here, returns { value: 3, done: false }
  // implicit return → { value: undefined, done: true }
}

// Calling the generator creates a generator object — body does NOT run yet
const gen = threeValues();

console.log(gen.next()); // logs "start", returns { value: 1, done: false }
console.log(gen.next()); // logs "middle", returns { value: 2, done: false }
console.log(gen.next()); // logs "end", returns { value: 3, done: false }
console.log(gen.next()); // returns { value: undefined, done: true }

// Generators are iterable — use for...of directly
for (const v of threeValues()) {
  console.log(v); // 1, 2, 3
}
The pause-and-resume mental model Imagine the generator function is a conversation. yield is you saying "here's a value — I'll wait for you to ask for the next one." .next() is the caller saying "okay, continue." The generator remembers exactly where it was — its local variables, its position in the code — and picks up right from that point.

6. Practical Generators — Lazy Evaluation

The most powerful use of generators: producing values on demand instead of computing them all upfront. This is called lazy evaluation.

// Infinite sequence — computing this all at once is impossible
function* integers() {
  let i = 0;
  while (true) { // infinite loop — fine! we only pull what we need
    yield i++;
  }
}

// Take just the first 5 values from an infinite sequence
function take(n, iterable) {
  const result = [];
  for (const value of iterable) {
    result.push(value);
    if (result.length === n) break;
  }
  return result;
}

console.log(take(5, integers())); // [0, 1, 2, 3, 4]

// Rewriting the range example from Section 4 — much simpler with a generator
function* range(from, to) {
  for (let i = from; i <= to; i++) {
    yield i;
  }
}

console.log([...range(1, 5)]); // [1, 2, 3, 4, 5]

// Paginatation: yield one page at a time
async function* fetchPages(baseUrl) {
  let page = 1;
  while (true) {
    const res = await fetch(`${baseUrl}?page=${page}`);
    const data = await res.json();
    if (data.length === 0) return; // no more pages
    yield data; // yield this page's results
    page++;
  }
}

// Consume pages with for-await-of
for await (const page of fetchPages('/api/items')) {
  console.log('Processing page:', page);
}

7. Generator Use Cases & Summary

Generators shine in situations that involve sequences of values produced over time:

Use CaseWhy a Generator Fits
Paginated API requestsYield one page at a time, stop when done — caller drives the pace
Infinite sequences (IDs, timestamps)Generate values lazily — never compute more than needed
State machinesEach yield represents a state; .next() drives transitions
Custom iterablesMuch cleaner than manually implementing the protocol
Async iteration (for await...of)Process async streams item by item without loading all into memory
 
// Summary: the chain of concepts

// 1. Symbol — unique, non-string key
const key = Symbol('myKey');

// 2. Symbol.iterator — well-known symbol that enables iteration
// 3. Iterable — has [Symbol.iterator]() → iterator
// 4. Iterator — has .next() → { value, done }
// 5. Generator — sugar that implements 3 + 4 automatically

function* colors() {
  yield 'red';
  yield 'green';
  yield 'blue';
}

// The generator object IS an iterator (has .next())
// AND an iterable (has [Symbol.iterator] that returns itself)
const gen = colors();
console.log(gen[Symbol.iterator]() === gen); // true
Interview insight When asked about generators in interviews, the key insight to communicate is lazy evaluation: "Values are produced on demand, which means you can represent infinite sequences and save memory by never computing values you don't actually use." Then mention async generators and pagination as a real-world application.
