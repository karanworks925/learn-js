The this Keyword

this is determined at call time — 4 rules, zero exceptions.

📖 Primary Sources for this lesson Read these after the lesson: javascript.info — this in methods and javascript.info — bind. Together they cover everything in this lesson with interactive examples.

1. What is this?

this is a special keyword that refers to an object. Which object depends entirely on how the function is called — not where the function is defined. This is the most important thing to internalize about this.

This contrasts directly with closures. Recall: a closure's scope is set at definition time. But this is set at call time. There are exactly 4 rules that determine the value of this. Learn the rules and you will never be confused by this again.

The core question When you see this inside a function, ask: "How is this function being called?" The answer to that question tells you what this is. Not where the function is defined — how it is called.

2. Rule 1 — Default Binding

When a function is called as a plain standalone function (not as a method, not with new, not with call/apply/bind), the default binding applies.

  • In non-strict mode: this = the global object (window in browsers, global in Node.js)
  • In strict mode: this = undefined
function whoAmI() {
  console.log(this);
}

// Non-strict mode
whoAmI(); // window (browser) or global (Node)

// Strict mode
"use strict";
function strictWhoAmI() {
  console.log(this); // undefined
}
strictWhoAmI();

// The danger — accidentally writing to window:
function dangerous() {
  this.name = "oops"; // In non-strict mode: sets window.name!
}
dangerous();
console.log(window.name); // "oops" — global pollution

3. Rule 2 — Implicit Binding

When a function is called as a method on an object (obj.method()), this is the object to the left of the dot at the call site. This is the most common binding.

const user = {
  name: "Karan",
  greet() {
    console.log(`Hello, I'm ${this.name}`);
  }
};

user.greet(); // "Hello, I'm Karan" — this = user (left of the dot)

Losing Implicit Binding — The Classic Bug

Implicit binding is fragile. As soon as you detach the method from its object, the binding is lost.

const user = {
  name: "Karan",
  greet() {
    console.log(`Hello, I'm ${this.name}`);
  }
};

// Detaching the method — binding is lost
const detachedGreet = user.greet;
detachedGreet(); // "Hello, I'm undefined" (or TypeError in strict mode)
                  // this = window/undefined — NOT user!

// This happens constantly with callbacks:
setTimeout(user.greet, 1000); // Also loses binding!
// setTimeout calls the function, not user.greet() — so this is not user
Most common this bug in production Passing a method as a callback always loses implicit binding. This breaks React class component methods, event handlers, and timer callbacks. The fixes are Rule 3 (bind) and Rule 6 (arrow functions).

4. Rule 3 — Explicit Binding

You can explicitly set this for any function call using three built-in methods: call, apply, and bind.

function introduce(greeting, punctuation) {
  console.log(`${greeting}, I'm ${this.name}${punctuation}`);
}

const person = { name: "Karan" };

// call — invoke immediately, pass args one by one
introduce.call(person, "Hello", "!");
// "Hello, I'm Karan!"

// apply — invoke immediately, pass args as an array
introduce.apply(person, ["Hi", "."]);
// "Hi, I'm Karan."

// bind — does NOT invoke. Returns a NEW function with this permanently locked to person
const boundIntroduce = introduce.bind(person);
boundIntroduce("Hey", "?");
// "Hey, I'm Karan?"

// bind also supports partial application (pre-filling arguments)
const formalIntroduce = introduce.bind(person, "Good day");
formalIntroduce(".");
// "Good day, I'm Karan."

5. Rule 4 — new Binding

When you call a function with the new keyword, JavaScript does something special behind the scenes — it creates a brand new empty object, sets this to that object inside the function, and returns it automatically (unless the function explicitly returns a different object).

function Person(name, age) {
  // When called with new:
  // 1. A new empty object is created: {}
  // 2. this is set to that object
  this.name = name;
  this.age = age;
  // 3. The object is returned automatically
}

const karan = new Person("Karan", 22);
console.log(karan.name); // "Karan"
console.log(karan.age);  // 22

// Without new — this is the global object/undefined: BUG
const oops = Person("Oops", 0); // this = window, returns undefined
Convention: capitalize constructor functions By convention, functions meant to be called with new start with an uppercase letter (Person, Animal). This is a visual warning: "call me with new". If you see a capitalized function called without new, that's a bug.

6. Arrow Functions — No Own this

Arrow functions do not have their own this. Instead, they inherit this from the surrounding lexical scope — wherever the arrow function was written. This is the only exception to the "call time determines this" rule.

This makes arrow functions perfect as callbacks inside methods — they "capture" the outer this at definition time.

// THE BUG — regular function callback loses this
const timer = {
  count: 0,
  start() {
    setTimeout(function() {
      this.count++; // this = window (or undefined in strict) — NOT timer
      console.log(this.count); // NaN or error
    }, 1000);
  }
};

// THE FIX — arrow function inherits this from start()'s lexical scope
const timer2 = {
  count: 0,
  start() {
    // 'this' here is timer2 (implicit binding from timer2.start())
    setTimeout(() => {
      this.count++; // ✅ arrow inherits this from start() — this is timer2
      console.log(this.count); // 1
    }, 1000);
  }
};

timer2.start();
Never use arrow functions as object methods Because arrow functions don't have their own this, using one as an object method means this won't be the object — it will be whatever this was in the scope where the object was defined (usually window). 
// WRONG — arrow as method: this is NOT the object
const bad = {
  name: "Karan",
  greet: () => {
    console.log(this.name); // undefined — this is the outer scope (window/global)
  }
};
bad.greet(); // undefined

// CORRECT — regular function as method
const good = {
  name: "Karan",
  greet() {
    console.log(this.name); // "Karan" — this is the object
  }
};
good.greet(); // "Karan"

7. Priority Order

When multiple rules could apply, there is a clear priority order. Higher number = higher priority.

PriorityRuleHow to recognize itWhat this is
4 (highest)new bindingnew Fn()The newly created object
3Explicit bindingfn.call(obj), fn.apply(obj), fn.bind(obj)()The first argument to call/apply/bind
2Implicit bindingobj.fn()The object to the left of the dot
1 (lowest)Default bindingfn() — plain callwindow (non-strict) or undefined (strict)

Arrow functions are not in the table because they do not participate in any of these rules — they always inherit this lexically and it cannot be overridden by any of the above rules (not even call, apply, or bind).

// Demonstrating priority — explicit beats implicit
const obj1 = { name: "obj1", greet() { return this.name; } };
const obj2 = { name: "obj2" };

console.log(obj1.greet());               // "obj1" — implicit binding
console.log(obj1.greet.call(obj2));      // "obj2" — explicit beats implicit

// Arrow function cannot be rebound
const arrowFn = () => this;
console.log(arrowFn.call({ name: "force" })); // Still the outer this — call is ignored
