Operators & Control Flow

The decision-making machinery of every JavaScript program.

1. Comparison Operators

JavaScript has two ways to check equality. Getting this wrong is one of the most common sources of bugs in real codebases.

OperatorNameType coercion?ExampleResult
==Loose equalityYes5 == "5"true
===Strict equalityNo5 === "5"false
!=Loose inequalityYes5 != "5"false
!==Strict inequalityNo5 !== "5"true
<, >, <=, >=RelationalYes"10" > 9true

When == compares different types, it converts them to numbers first — then compares. This leads to deeply surprising results:

// The == traps that trip up developers
0 == ""          // true  — both convert to 0
0 == "0"         // true  — "0" converts to 0
"" == "0"        // false — string to string, not equal
null == undefined  // true  — special case, only these two are == to each other
null == 0         // false — null only equals null or undefined via ==
false == "0"     // true  — both convert to 0

// === never coerces — always compare type AND value
5 === "5"         // false — different types
null === undefined // false — different types
Interview Trap null == undefined is true with loose equality — this is a special rule in the spec. But null == 0 is false. null only loosely equals null and undefined, nothing else.

String comparisons use Unicode code points, which can surprise you with numbers-as-strings:

// String comparison: lexicographic (character by character)
"apple" < "banana"   // true  — 'a' < 'b'
"Z" < "a"            // true  — uppercase letters come before lowercase in Unicode

// Comparing a string to a number: the string is coerced to number
"10" > 9            // true  — "10" becomes 10
"10" > "9"          // false — "1" < "9" (string comparison, not numeric!)

2. Logical Operators

Logical operators in JavaScript do not just return true or false. They return one of the actual operand values. This is crucial to understand for real-world code.

Short-Circuit Evaluation

JS evaluates the left side first. If the result is already determined, it stops and returns that value — it never looks at the right side.

// && (AND): returns the FIRST FALSY value, or the LAST value if all are truthy
true && "hello"       // "hello"   — left is truthy, so returns right
false && "hello"      // false     — left is falsy, short-circuits immediately
0 && doSomething()   // 0         — doSomething() is NEVER called
"user" && "logged in" // "logged in" — both truthy, returns last

// || (OR): returns the FIRST TRUTHY value, or the LAST value if all are falsy
"hello" || "default" // "hello"   — left is truthy, short-circuits
"" || "default"      // "default" — left is falsy, evaluates right
0 || 42              // 42        — 0 is falsy, returns 42
null || undefined    // undefined — both falsy, returns the last one
// Practical uses of short-circuit evaluation
const user = getUser() || { name: "Guest" }; // default if null/falsy

// Guard: only call save() if form is valid
isValid && save();

// Logical assignment operators (ES2021)
let count = 0;
count ||= 10;    // count = count || 10 → count is now 10 (was falsy)

let name = "Karan";
name &&= name.toUpperCase(); // Only assigns if name is truthy → "KARAN"

let config = null;
config ??= { debug: false }; // Only assigns if config is null or undefined
Key Insight ||= and &&= short-circuit too — the right side only runs when needed. This matters if the right side has side effects (like calling a function or making a request).

3. The Nullish Coalescing Operator ??

?? (introduced in ES2020) solves a real problem: || treats 0, "", and false as falsy, which means it replaces them with defaults even when those values are intentional.

// Problem with || for defaults
const volume = 0;
const level = volume || 50; // 50 — WRONG! 0 is a valid volume setting

// Solution: ?? only triggers on null or undefined
const level2 = volume ?? 50; // 0  — CORRECT! 0 is not null/undefined

// More examples
null ?? "default"      // "default"  — null triggers ??
undefined ?? "default" // "default"  — undefined triggers ??
0 ?? "default"         // 0          — 0 does NOT trigger ??
"" ?? "default"        // ""         — empty string does NOT trigger ??
false ?? "default"    // false      — false does NOT trigger ??

Optional Chaining ?.

Access properties deep in an object without throwing if something in the chain is null or undefined.

const user = { profile: { address: null } };

// Without optional chaining — throws TypeError
console.log(user.profile.address.city); // ❌ TypeError: Cannot read properties of null

// With optional chaining — safely returns undefined
console.log(user?.profile?.address?.city); // undefined — no error

// Works with methods too
user?.profile?.getAvatar(); // undefined if getAvatar doesn't exist

// Works with arrays
const arr = null;
arr?.[0]; // undefined — doesn't throw

// Combine ?? and ?. for clean defaults
const city = user?.profile?.address?.city ?? "Unknown City";
Pro Pattern Combining ?. and ?? is extremely common in modern JS. It replaces what used to take 5 lines of null checks. const city = user?.address?.city ?? "Unknown";

4. Ternary Operator

The ternary is a compact if/else for expressions — useful when you need a value based on a condition.

// Basic ternary
const age = 20;
const status = age >= 18 ? "adult" : "minor";
console.log(status); // "adult"

// Useful inline in JSX / template literals
const label = isLoggedIn ? "Log Out" : "Log In";

// Nested ternaries — readable only when kept to 2 levels max
const score = 72;
const grade = score >= 90 ? "A"
             : score >= 80 ? "B"
             : score >= 70 ? "C"
             : "F";
// grade is "C"
Warning Deeply nested ternaries become unreadable fast. If you need more than two levels, switch to if/else if or a lookup object. Code is read more often than it is written.

5. if / else if / else

The workhorse of control flow. Simple to use — but knowing when to switch to another pattern is the mark of an experienced developer.

// Basic if/else
const hour = new Date().getHours();

if (hour < 12) {
  console.log("Good morning");
} else if (hour < 17) {
  console.log("Good afternoon");
} else {
  console.log("Good evening");
}

// Truthy/falsy checks are common
const username = getUsername();
if (username) {          // truthy check — catches null, undefined, ""
  greet(username);
} else {
  showLoginPrompt();
}

// Early return pattern — flattens nesting (preferred in real code)
function processPayment(amount) {
  if (amount <= 0) return "Invalid amount";
  if (!isAuthenticated()) return "Not logged in";
  if (!hasFunds(amount)) return "Insufficient funds";

  // Happy path — no deep nesting
  return charge(amount);
}
Senior Dev Pattern The early return pattern (also called "guard clauses") is preferred over deeply nested if/else. Handle edge cases first and return early. The main logic stays at the left margin.

6. switch / case

switch shines when you're comparing one value against many possible fixed values. It uses === (strict equality) internally — no type coercion.

const day = "Monday";

switch (day) {
  case "Monday":
  case "Tuesday":
  case "Wednesday":
  case "Thursday":
  case "Friday":
    console.log("Weekday"); // ← all 5 cases fall through to here
    break;                  // ← STOP. Without this, execution continues!
  case "Saturday":
  case "Sunday":
    console.log("Weekend");
    break;
  default:                  // ← runs if no case matched
    console.log("Unknown day");
}

The Fall-Through Trap

// What happens WITHOUT break — fall-through
const x = 1;
switch (x) {
  case 1:
    console.log("one");   // ← this runs
  case 2:
    console.log("two");   // ← this ALSO runs (no break above!)
  case 3:
    console.log("three"); // ← this ALSO runs
}
// Output: "one", "two", "three"

// Intentional fall-through (grouped cases) is fine and readable
// Accidental fall-through (forgetting break) is a very common bug
Warning Always add break at the end of each case unless you intentionally want fall-through. If you do want fall-through, add a comment: // intentional fall-through.
