async/await

Async/await makes asynchronous code read like synchronous code — but under the hood it's still Promises.

Primary Source for this lesson Read after the lesson: javascript.info — async/await. This builds directly on the Promises lesson — if anything here is unclear, go back to Lesson 18 first.

1. async Functions

The async keyword before a function declaration does two things:

  1. It makes the function always return a Promise — no matter what you return inside
  2. It enables the use of await inside the function body
// Returning a plain value — automatically wrapped in Promise.resolve()
async function getNumber() {
  return 42;
}

getNumber().then(n => console.log(n)); // 42
// Same as: Promise.resolve(42).then(...)

// Returning a Promise — the function's Promise adopts its state
async function fetchData() {
  return fetch('https://api.example.com/data'); // fetch returns a Promise
}
// The returned Promise waits for fetch to resolve

// Throwing inside an async function — rejects the returned Promise
async function fail() {
  throw new Error("Something went wrong");
}
fail().catch(err => console.error(err.message)); // "Something went wrong"
async Functions Are Always Async Even if the body of an async function is completely synchronous, the return value is still wrapped in a Promise. async function f() {'{'} return 1; {'}'} returns Promise{'<'}1{'>'}. The caller always gets a Promise.

2. The await Keyword

await can only be used inside an async function (or at the top level of an ES Module). It does one thing: pauses the execution of the current async function until the Promise settles, then returns the resolved value.

async function loadUser() {
  console.log("Fetching user...");

  // Execution PAUSES here until the fetch Promise resolves.
  // The event loop is NOT blocked — other code continues running.
  const response = await fetch('https://api.example.com/user/1');

  // This line only runs after the fetch completes
  const user = await response.json();

  console.log("Got user:", user.name);
  return user;
}

loadUser(); // starts the async function
console.log("This runs immediately while loadUser is waiting");

The critical thing to understand: await pauses only the async function it lives in. The rest of the JavaScript program keeps running. The event loop continues picking up other tasks. This is not blocking.

await Outside async is a SyntaxError You cannot use await inside a regular function, an arrow function that is not async, or at the top level of a CommonJS module. The only exceptions are: async functions and ES Module top-level (ES2022+).

3. Error Handling with try/catch

When an awaited Promise rejects, await throws that rejection as an error — which means you can catch it with a standard try/catch block. This is one of the biggest readability improvements over .catch() chaining.

// Promise .catch() style — catching at the end
fetchUser(id)
  .then(user => processUser(user))
  .then(result => saveResult(result))
  .catch(err => console.error("Failed:", err));

// async/await style — reads like synchronous error handling
async function run() {
  try {
    const user = await fetchUser(id);
    const result = await processUser(user);
    await saveResult(result);
  } catch (err) {
    console.error("Failed:", err);
    // All three errors caught in one place
  } finally {
    hideSpinner(); // always runs
  }
}
Granular Error Handling You can wrap individual await expressions in their own try/catch blocks when different errors need different handling. Or use the pattern const [err, result] = await somePromise.then(v => [null, v]).catch(e => [e, null]) to avoid nesting entirely (sometimes called the "go-style" pattern).

4. Sequential vs Parallel Execution

This is one of the most important performance topics in async JavaScript, and one of the most common interview questions. Getting it wrong means slow code in production.

Sequential — Each Waits for the Previous

// BAD for independent operations — total time = t1 + t2 + t3
async function loadDashboard() {
  const user     = await fetchUser();     // waits 300ms
  const orders   = await fetchOrders();   // waits another 400ms
  const settings = await fetchSettings(); // waits another 200ms
  // Total: ~900ms — but these three are independent!
  return { user, orders, settings };
}

Parallel — Start All, Wait for All

// GOOD — total time = max(t1, t2, t3) = ~400ms
async function loadDashboard() {
  // Start all three at the same time
  const [user, orders, settings] = await Promise.all([
    fetchUser(),
    fetchOrders(),
    fetchSettings(),
  ]);
  return { user, orders, settings };
}
The Sequential Trap is Extremely Common Writing const a = await p1; const b = await p2; when p1 and p2 are independent is one of the most frequent performance bugs in real-world async code. In a large app, this can add seconds to page load time. Always ask: "Do these operations depend on each other?" If not, use Promise.all().

Manual Parallel Start (without Promise.all)

// Another valid pattern: start Promises first, await later
async function loadDashboard() {
  const userPromise     = fetchUser();     // starts immediately
  const ordersPromise   = fetchOrders();   // starts immediately
  const settingsPromise = fetchSettings(); // starts immediately

  const user     = await userPromise;     // now we wait
  const orders   = await ordersPromise;
  const settings = await settingsPromise;

  return { user, orders, settings };
}

5. async/await vs Promises — When to Use Each

ScenarioRecommended approach
Sequential async steps with clear dependenciesasync/await — cleaner and easier to read
Running multiple Promises in parallelPromise.all() / Promise.allSettled() (mix with await)
Racing Promises or getting first successPromise.race() / Promise.any()
Wrapping a callback API into a Promisenew Promise()
Event handlers (onClick, addEventListener)Can be async — just be careful with unhandled rejections
They Are Not Competitors async/await and Promises work together. await is syntactic sugar over .then() — it compiles to Promise chains under the hood. Mixing both is common, correct, and often the clearest approach. For example: const [a, b] = await Promise.all([p1, p2]) uses both.

6. Common Mistakes

Mistake 1 — Forgetting await

async function getUser() {
  const user = fetchUser(); // ❌ forgot await — user is a Promise, not the value!
  console.log(user.name);   // undefined — Promise objects have no .name property
}

// Fix:
const user = await fetchUser(); // ✅

Mistake 2 — Sequential When Parallel is Possible

// ❌ slow — see Section 4 above for the fix
const a = await fetchA();
const b = await fetchB(); // waits for A before starting B

Mistake 3 — Not Handling Errors

// ❌ unhandled rejection — will crash the process in Node.js
async function run() {
  const data = await mightFail(); // if this rejects, nothing catches it
}
run(); // missing .catch() here!

// ✅ always handle rejections
run().catch(console.error);
// or use try/catch inside the async function

Mistake 4 — await Inside forEach

// ❌ WRONG — forEach does not await async callbacks
async function processAll(items) {
  items.forEach(async (item) => {
    await process(item); // the await is inside the callback, not processAll
  });
  // processAll finishes immediately — items are NOT fully processed!
}

// ✅ CORRECT — use for...of, which respects await
async function processAll(items) {
  for (const item of items) {
    await process(item); // sequential — each waits for the previous
  }
}

// ✅ ALSO CORRECT — if order doesn't matter, run in parallel
async function processAll(items) {
  await Promise.all(items.map(item => process(item)));
}
The forEach Mistake is Extremely Common This trips up developers at every level. Array.forEach() is not async-aware. It fires each callback but does not wait for the returned Promise to settle. The function appears to "finish" before any async work is done. Use for...of for sequential, or Promise.all(arr.map(...)) for parallel.

7. Async Iteration

When data arrives in chunks over time (streams, pagination, WebSockets), you can use for await...of to consume async iterables — values that arrive asynchronously one at a time.

// Async generator — yields values asynchronously
async function* fetchPages(url) {
  let page = 1;
  while (true) {
    const data = await fetch(`${url}?page=${page}`).then(r => r.json());
    if (!data.results.length) break;
    yield data.results;
    page++;
  }
}

// Consuming an async iterable with for await...of
async function getAllResults() {
  for await (const page of fetchPages('https://api.example.com/items')) {
    console.log("Got page:", page);
    // waits for the next page before continuing
  }
}

Node.js streams, browser ReadableStream, and many database cursors are async iterables. for await...of is the clean, modern way to consume them.
