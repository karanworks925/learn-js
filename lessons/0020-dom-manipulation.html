DOM Manipulation

The DOM is your interface to the browser — learn to read and rewrite it.

Primary Source for this lesson Read after the lesson: javascript.info — DOM nodes. Then continue with Searching the DOM and Modifying the Document.

1. What is the DOM?

When the browser loads an HTML file, it parses it and builds an in-memory tree of objects called the Document Object Model (DOM). Every HTML element, attribute, and piece of text becomes a node in this tree. JavaScript can read and modify this tree, which is how web pages become interactive.

<!-- This HTML -->
<div id="app">
  <h1>Hello</h1>
  <p class="intro">World</p>
</div>

// Becomes this tree:
// document
//   └── html
//         └── body
//               └── div#app
//                     ├── h1 → "Hello"
//                     └── p.intro → "World"

The document object is the entry point to the DOM from JavaScript. Everything hangs off document.

Live vs Static Collections

Some DOM query methods return live collections that automatically update when the DOM changes. Others return static snapshots that do not update. Knowing which is which prevents subtle bugs.

MethodReturnsLive?
getElementById()Element or nullN/A (single element)
querySelector()Element or nullN/A (single element)
querySelectorAll()NodeListNo — static snapshot
getElementsByClassName()HTMLCollectionYes — updates automatically
getElementsByTagName()HTMLCollectionYes — updates automatically
element.childrenHTMLCollectionYes — updates automatically

2. Selecting Elements

Modern JavaScript code almost exclusively uses querySelector and querySelectorAll. They accept any valid CSS selector, making them extremely flexible.

// getElementById — fastest, by ID only, returns element or null
const app = document.getElementById('app');

// querySelector — returns FIRST match of any CSS selector
const heading  = document.querySelector('h1');
const firstBtn = document.querySelector('.btn.primary');
const navLink  = document.querySelector('nav > a[href="/home"]');

// querySelectorAll — returns ALL matches as a static NodeList
const items = document.querySelectorAll('.list-item');
items.forEach(item => console.log(item.textContent));

// Scoped query — search within an element, not the whole document
const nav  = document.querySelector('nav');
const link = nav.querySelector('a'); // finds first  inside nav only
Always Check for null querySelector() returns null if no match is found. Calling methods on null throws a TypeError. Always check: const el = document.querySelector('.thing'); if (el) { ... } — or use optional chaining: document.querySelector('.thing')?.classList.add('active').

3. Reading & Modifying Content

There are three main ways to get or set the content of an element, and choosing the wrong one can introduce security vulnerabilities.

const el = document.querySelector('.message');

// textContent — plain text, SAFE
console.log(el.textContent);       // read
el.textContent = "Updated text";   // write — HTML is escaped automatically

// innerHTML — parses as HTML, DANGEROUS with user input
console.log(el.innerHTML);         // read — includes HTML tags
el.innerHTML = "<strong>Bold</strong>"; // write — parsed as HTML

// innerText — rendered text only (respects CSS visibility)
// Triggers a layout calculation — slower than textContent
console.log(el.innerText);         // read
innerHTML + User Input = XSS Never set innerHTML to a string that contains user-provided data. A user could inject <script>stealCookies()</script> or <img src="x" onerror="stealData()">. Use textContent for text, or DOMPurify if you must render user-provided HTML.

Attributes and Data Attributes

const link = document.querySelector('a');

// Standard attributes
link.getAttribute('href');              // read attribute value
link.setAttribute('href', '/about');   // set attribute
link.removeAttribute('target');        // remove attribute
link.hasAttribute('disabled');         // check — returns boolean

// data-* attributes via .dataset — camelCase conversion
// <div data-user-id="42" data-role="admin">
const card = document.querySelector('.user-card');
console.log(card.dataset.userId); // "42" — data-user-id becomes userId
console.log(card.dataset.role);   // "admin"
card.dataset.verified = "true";   // adds data-verified="true" to the DOM

4. Modifying Styles & Classes

Avoid setting inline styles directly when possible — prefer toggling CSS classes so your styling stays in CSS where it belongs. Inline styles have the highest specificity and are hard to override.

const box = document.querySelector('.box');

// Inline styles — use camelCase for CSS property names
box.style.color           = 'red';
box.style.backgroundColor = '#f0f0f0'; // background-color → backgroundColor
box.style.fontSize        = '1.5rem';

// classList — the preferred way
box.classList.add('active');          // adds class
box.classList.remove('hidden');       // removes class
box.classList.toggle('open');         // adds if absent, removes if present
box.classList.toggle('open', true);  // force-add (second arg controls state)
box.classList.contains('active');    // returns true/false
box.classList.replace('old', 'new'); // replace one class with another

// Add multiple classes at once
box.classList.add('card', 'card--primary', 'visible');
Reading Computed Styles element.style only shows inline styles — not styles applied by CSS classes or stylesheets. To read the actual computed value of any CSS property, use window.getComputedStyle(element).getPropertyValue('color').

5. Creating & Adding Elements

Creating elements in JavaScript involves three steps: create the element, configure it, then insert it into the document. Elements that are not in the document are not visible — they exist in memory only.

// Step 1: Create
const card = document.createElement('div');

// Step 2: Configure
card.className   = 'card';
card.textContent = 'Hello from JS!';
card.setAttribute('data-id', '123');

// Step 3: Insert into the document
const container = document.querySelector('.container');

container.append(card);          // add at END — accepts strings and nodes
container.prepend(card);         // add at START
container.append("plain text");  // strings are inserted as text nodes (safe)

card.before("<-- here");        // insert before card as sibling
card.after("here -->");          // insert after card as sibling

card.remove();                    // remove element from the DOM

Older insertBefore API

// insertBefore — older API, still common in legacy code
// parent.insertBefore(newNode, referenceNode)
// inserts newNode immediately before referenceNode
const list = document.querySelector('ul');
const newItem = document.createElement('li');
newItem.textContent = 'New first item';
list.insertBefore(newItem, list.firstElementChild); // insert at start
append() vs appendChild() append() is the modern method: it accepts multiple arguments, can insert strings (as text nodes), and returns nothing. appendChild() is the older method: accepts only a single Node, returns the inserted node. In new code, prefer append().

6. Traversing the DOM

Once you have a reference to one element, you can navigate to related elements using traversal properties. All properties listed below skip text nodes — they return only element nodes.

const el = document.querySelector('.item');

// Navigate UP
el.parentElement;                // direct parent element
el.closest('.container');       // walk UP tree — first ancestor matching selector

// Navigate DOWN
el.children;                     // live HTMLCollection of child elements
el.firstElementChild;            // first child element
el.lastElementChild;             // last child element

// Navigate SIDEWAYS
el.nextElementSibling;           // next sibling element
el.previousElementSibling;      // previous sibling element
// Practical example — event delegation using .closest()
document.querySelector('.list')
  .addEventListener('click', (e) => {
    // User might click a child of .list-item (e.g., a span inside it)
    // .closest() walks up the tree to find the .list-item ancestor
    const item = e.target.closest('.list-item');
    if (!item) return; // click was outside a list item
    console.log("Clicked item:", item.dataset.id);
  });
element.closest() is Extremely Useful closest() starts at the element itself and walks up the tree, returning the first element that matches the selector. It returns null if nothing matches. This is the foundation of event delegation — attaching one listener to a parent instead of many listeners to children.

7. Cloning, DocumentFragment & Performance

DOM manipulation is expensive because changing the DOM can trigger reflow (the browser recalculates layout) and repaint (the browser redraws pixels). Each is slow. The goal is to minimize how many times you touch the live DOM.

Cloning Nodes

const template = document.querySelector('.card-template');

// cloneNode(false) — shallow clone (no children)
const shallowCopy = template.cloneNode(false);

// cloneNode(true) — deep clone (includes all descendants)
const deepCopy = template.cloneNode(true);
deepCopy.querySelector('h2').textContent = "New Card Title";
document.querySelector('.grid').append(deepCopy);

DocumentFragment — Batch DOM Updates

// BAD — inserts into live DOM 100 times, triggers 100 reflows
const list = document.querySelector('ul');
for (let i = 0; i < 100; i++) {
  const li = document.createElement('li');
  li.textContent = `Item ${i}`;
  list.append(li); // live DOM insertion each time
}

// GOOD — build off-screen, insert once, one reflow
const fragment = document.createDocumentFragment();
for (let i = 0; i < 100; i++) {
  const li = document.createElement('li');
  li.textContent = `Item ${i}`;
  fragment.append(li); // off-tree — no reflow
}
list.append(fragment); // ONE insertion — one reflow

A DocumentFragment is a lightweight container that lives outside the document tree. You can build complex DOM structures inside it, then insert everything at once. When inserted, the fragment itself disappears — only its children are added.

Modern Frameworks Abstract This React, Vue, and Svelte use virtual DOMs or compiled templates to batch DOM updates automatically. But understanding manual DOM manipulation is still essential — for debugging, for building without frameworks, for performance analysis, and for interviews.
