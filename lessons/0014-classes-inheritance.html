ES6 Classes & Inheritance

Classes are syntactic sugar — but knowing what they do under the hood separates seniors from juniors.

Primary Source for this lesson Read this after the lesson: javascript.info — Classes. The sections on private fields and the internal mechanics are especially valuable for interviews.

1. Class Syntax

The class keyword, introduced in ES6, is syntactic sugar over the prototype system you learned in Lesson 13. Under the hood, a class is still a function with a prototype object. Classes make the pattern cleaner and harder to misuse — but they don't introduce a new inheritance model.

class Person {
  // constructor is called when you use 'new'
  constructor(name, age) {
    this.name = name; // own instance property
    this.age = age;
  }

  // Methods are placed on Person.prototype — NOT on each instance
  greet() {
    return `Hi, I'm ${this.name} and I'm ${this.age}`;
  }

  birthday() {
    this.age++;
  }
}

const karan = new Person("Karan", 24);
karan.greet(); // "Hi, I'm Karan and I'm 24"
karan.birthday();
karan.age;     // 25

// Under the hood — exactly equivalent to:
function Person(name, age) { this.name = name; this.age = age; }
Person.prototype.greet = function() { ... };

2. Getters and Setters

Getters and setters let you define computed properties — they look like regular property accesses but secretly run functions. Use them to compute derived values or to validate before setting.

class User {
  constructor(firstName, lastName) {
    this.firstName = firstName;
    this.lastName = lastName;
    this._age = 0; // convention: _ means "meant to be private"
  }

  // Getter — accessed as a property, not a call
  get fullName() {
    return `${this.firstName} ${this.lastName}`;
  }

  // Setter — validates before storing
  set age(value) {
    if (value < 0 || value > 150) {
      throw new Error("Invalid age");
    }
    this._age = value;
  }

  get age() {
    return this._age;
  }
}

const user = new User("Karan", "Mourya");

user.fullName;     // "Karan Mourya" — getter called, looks like a property
user.age = 24;    // setter called — validates, then sets _age
user.age;          // 24 — getter called
user.age = -5;   // throws Error("Invalid age")

3. Static Methods and Properties

A static method or property belongs to the class itself, not to any instance. You call it on the class name, never on an instance. Static methods are commonly used for factory functions, utility helpers, and caching.

class MathHelper {
  static PI = 3.14159;

  static circleArea(r) {
    return MathHelper.PI * r * r;
  }

  static clamp(value, min, max) {
    return Math.min(Math.max(value, min), max);
  }
}

MathHelper.circleArea(5);    // 78.53975
MathHelper.clamp(150, 0, 100); // 100

// Cannot call on an instance:
const m = new MathHelper();
m.circleArea(5); // TypeError: m.circleArea is not a function

// Factory method pattern — a common use of static
class Color {
  constructor(r, g, b) { this.r = r; this.g = g; this.b = b; }

  static fromHex(hex) {
    const r = parseInt(hex.slice(1, 3), 16);
    const g = parseInt(hex.slice(3, 5), 16);
    const b = parseInt(hex.slice(5, 7), 16);
    return new Color(r, g, b);
  }
}

const red = Color.fromHex("#ff0000"); // Color { r: 255, g: 0, b: 0 }

4. Inheritance with extends

The extends keyword sets up prototype chain inheritance between classes. A subclass inherits all methods of the parent class and can add its own or override existing ones.

class Animal {
  constructor(name) {
    this.name = name;
  }

  speak() {
    return `${this.name} makes a sound`;
  }
}

class Dog extends Animal {
  constructor(name, breed) {
    super(name); // MUST call super() before using 'this'
    this.breed = breed;
  }

  // Method overriding — Dog provides its own speak()
  speak() {
    return `${this.name} barks`;
  }

  describe() {
    // super.speak() calls the parent's version
    return `${super.speak()} and is a ${this.breed}`;
  }
}

const rex = new Dog("Rex", "Labrador");
rex.speak();    // "Rex barks" — overridden method
rex.describe(); // "Rex makes a sound and is a Labrador"

// Prototype chain: rex → Dog.prototype → Animal.prototype → Object.prototype → null
super() is mandatory in subclass constructors If your subclass has a constructor, you MUST call super() before any reference to this. If you omit it, you get a ReferenceError: "Must call super constructor in derived class before accessing 'this'". This is one of the most common class-related errors.

5. Private Fields and Methods

The # prefix creates a truly private field or method (ES2022). Unlike the old _name convention, private fields are enforced by the language — they cannot be accessed or even seen from outside the class body.

class BankAccount {
  // Private fields must be declared at the top of the class body
  #balance = 0;
  #owner;

  constructor(owner, initialBalance) {
    this.#owner = owner;
    this.#balance = initialBalance;
  }

  // Private method
  #validate(amount) {
    if (amount <= 0) throw new Error("Amount must be positive");
  }

  deposit(amount) {
    this.#validate(amount); // can call private methods from inside
    this.#balance += amount;
  }

  withdraw(amount) {
    this.#validate(amount);
    if (amount > this.#balance) throw new Error("Insufficient funds");
    this.#balance -= amount;
  }

  get balance() { return this.#balance; } // read-only via getter
}

const acc = new BankAccount("Karan", 1000);
acc.deposit(500);
acc.balance; // 1500

acc.#balance; // SyntaxError: Private field '#balance' must be declared in an enclosing class
acc.#validate(100); // SyntaxError

Convention onlythis._balanceNo — still public, just a naming hint

6. Class vs Constructor Function — What's Different?

Classes are not just cosmetic. They have a few concrete behavioral differences from constructor functions:

Behaviorclassfunction constructor
HoistingIn TDZ — cannot use before declarationFully hoisted (function declaration)
Strict modeClass body is always strict modeDepends on surrounding scope
Method enumerabilityMethods are non-enumerable (don't show in for...in)Prototype methods are enumerable by default
Call without newTypeError — must use newSilently runs as a regular function
 
// Class is NOT hoisted like a function declaration
const p = new Person(); // ReferenceError — can't use before declaration
class Person {}

// But a function constructor IS hoisted:
const p2 = new OldPerson(); // Works fine!
function OldPerson() {}

// Under the hood — classes still use prototypes
class Cat { meow() {} }
typeof Cat; // "function" — a class IS a function under the hood
Cat.prototype.meow; // the method lives on the prototype

7. instanceof

instanceof checks whether an object has a particular class (or constructor function) anywhere in its prototype chain. It's the standard way to type-check class instances.

class Animal {}
class Dog extends Animal {}
class Cat extends Animal {}

const rex = new Dog();

rex instanceof Dog;    // true
rex instanceof Animal; // true — Dog inherits from Animal
rex instanceof Cat;    // false
rex instanceof Object; // true — everything inherits from Object

// instanceof checks the prototype chain, not a tag or flag
// Chain: rex → Dog.prototype → Animal.prototype → Object.prototype → null
// Animal.prototype is in the chain → instanceof Animal is true
Use instanceof with custom errors too When you have a hierarchy of custom error classes, instanceof lets you catch exactly what you need. This is a real pattern covered in the next lesson on error handling.
