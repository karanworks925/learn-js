The Event Loop

JavaScript is single-threaded — the event loop is how it handles concurrency anyway.

Primary Sources for this lesson Read after the lesson: MDN — The Event Loop. Then watch: Philip Roberts — "What the heck is the event loop anyway?" (JSConf EU). It is the single best 26-minute investment you can make to understand this topic.
Why This Lesson Matters The event loop is the most commonly misunderstood concept in JavaScript. It comes up in almost every senior developer interview. More importantly, getting it wrong causes real bugs: race conditions, UI freezes, and promises resolving in unexpected orders. Take this lesson slowly.

1. JavaScript is Single-Threaded

JavaScript runs on a single thread. That means there is exactly one call stack, and exactly one thing executes at any moment. There is no parallel execution on the main JavaScript thread.

This sounds like it should make async work impossible. How can you wait for a network request if you can only do one thing at a time? The answer is the event loop — but first, we need to understand the call stack.

Common Misconception JavaScript itself is not asynchronous. The language is synchronous and single-threaded. Async behavior comes from the environment (browser or Node.js) providing APIs that run outside the JS engine and notify JS when they're done.

2. The Call Stack

The call stack is a LIFO (Last In, First Out) data structure that tracks which function is currently executing. Every time a function is called, a new frame is pushed onto the stack. When the function returns, its frame is popped off.

Walking Through 3 Nested Function Calls

function multiply(a, b) {
  return a * b;            // step 3: runs, returns 6, frame popped
}

function square(n) {
  return multiply(n, n);  // step 2: calls multiply → pushes multiply frame
}

function main() {
  const result = square(3); // step 1: calls square → pushes square frame
  console.log(result);      // step 4: square done, logs 9
}

main();

Here is the exact sequence of stack operations:

  1. main() is called → push main frame. Stack: [main]
  2. square(3) is called inside main → push square. Stack: [main, square]
  3. multiply(3, 3) is called inside square → push multiply. Stack: [main, square, multiply]
  4. multiply returns 9 → pop multiply. Stack: [main, square]
  5. square returns 9 → pop square. Stack: [main]
  6. console.log(9) executes. Stack briefly: [main, console.log] then pops.
  7. main returns → pop main. Stack: [] (empty)
Stack Overflow If a function calls itself infinitely (infinite recursion) with no base case, the stack grows until the browser throws a "Maximum call stack size exceeded" (RangeError). The stack has a finite size — typically a few thousand frames.

3. Web APIs — What Runs Outside the JS Engine

The browser (and Node.js) provides host APIs that run in separate threads or processes, completely outside the JavaScript engine. These include:

APIWhat it doesProvided by
setTimeout / setIntervalRuns a timerBrowser & Node.js
fetchMakes HTTP requestsBrowser & Node.js
DOM event listenersWaits for user interactionBrowser
fs.readFileReads files from diskNode.js
MutationObserverWatches for DOM changesBrowser

When you call setTimeout(fn, 1000), JavaScript hands the timer off to the Web API environment and continues executing other code immediately. After 1000ms, the Web API says "I'm done" and places fn into the callback queue.

The Key Mental Model Think of Web APIs as a separate "waiting room." JavaScript gives them a task and a callback, then walks away. When the task finishes, the Web API puts the callback into a queue. The event loop then decides when JavaScript picks it up.

4. The Callback Queue (Macrotask Queue)

The callback queue (also called the macrotask queue or task queue) holds callbacks that are ready to run. Callbacks arrive here from:

  • setTimeout and setInterval (after the timer expires)
  • DOM events (after a click, keypress, etc.)
  • fetch completion callbacks (via XMLHttpRequest)
  • I/O callbacks in Node.js

The critical rule: callbacks in the macrotask queue are only executed when the call stack is completely empty. If JavaScript is currently running synchronous code, no macrotask callback can interrupt it.

console.log("start");

setTimeout(() => {
  console.log("timeout"); // goes to callback queue
}, 0);

// Even though timeout is 0ms, this runs first:
for (let i = 0; i < 1000000; i++) {} // blocks the stack

console.log("end");
// Output: "start", "end", "timeout"
// The timeout callback waits for the stack to empty

5. The Microtask Queue

The microtask queue is a separate, higher-priority queue. Microtasks come from:

  • Promise .then(), .catch(), and .finally() callbacks
  • queueMicrotask(fn)
  • MutationObserver callbacks
  • await continuations (under the hood, these are Promises)

The microtask queue has a crucial priority rule: after every macrotask (including the initial script), ALL pending microtasks are drained completely before the next macrotask runs.

console.log("A");                              // sync

setTimeout(() => console.log("B"), 0);        // macrotask queue

Promise.resolve().then(() => console.log("C")); // microtask queue
Promise.resolve().then(() => console.log("D")); // microtask queue

console.log("E");                              // sync

// Output: A, E, C, D, B
Infinite Microtask Loop is Dangerous If a microtask keeps adding more microtasks, the macrotask queue is NEVER processed. The browser tab freezes. This is equivalent to an infinite loop for async code. Always be careful about recursively scheduling microtasks.

6. The Event Loop Algorithm

The event loop is a continuous algorithm running in the JavaScript environment. Here is its exact behavior, step by step:

  1. Execute the current synchronous script (this is the first "macrotask")
  2. Drain all microtasks — run every task in the microtask queue until it is empty. If a microtask adds more microtasks, run those too.
  3. Render (browser only) — the browser may update the screen here, between macrotasks
  4. Pick ONE macrotask from the callback queue and execute it
  5. Go back to step 2 — drain all microtasks again
  6. Repeat forever
Visual Mental Model Picture three boxes:
— Call Stack: what is running right now
— Microtask Queue: VIP waiting room — gets in next, always
— Macrotask Queue: regular waiting room — gets in only after the VIP room is empty

The event loop is the bouncer: it checks the stack (is it empty?), then clears the VIP room entirely, then lets one person from the regular room in, then repeats.

Why setTimeout(fn, 0) is Not Immediate

A timeout of 0 milliseconds does not mean "run now." It means "run after the current call stack is empty AND all pending microtasks have run." The minimum delay before a macrotask runs is always the time it takes for the current synchronous code and all microtasks to complete — which could be many milliseconds if there is heavy work happening.

7. The Classic Interview Question — Output Prediction

This exact question (or a variation) appears in nearly every senior JavaScript interview. Work through it yourself first, then read the step-by-step explanation.

console.log("1");

setTimeout(() => console.log("2"), 0);

Promise.resolve().then(() => console.log("3"));

console.log("4");

The answer is: 1, 4, 3, 2

Here is the precise step-by-step trace:

1
Line 1 — sync: console.log("1") executes immediately on the call stack. Output: "1"
2
Line 3 — setTimeout: JavaScript hands the timer to the Web API. The callback () => console.log("2") is registered. Since the delay is 0ms, the Web API immediately places it in the macrotask queue. The call stack does NOT wait.
3
Line 5 — Promise.resolve().then(): Promise.resolve() creates an already-resolved Promise. The .then() callback () => console.log("3") is immediately scheduled in the microtask queue. It does not run yet.
4
Line 7 — sync: console.log("4") executes immediately. Output: "4"
5
Script ends — call stack is now empty. The event loop kicks in. First: drain the microtask queue. The Promise callback runs: console.log("3"). Output: "3". Microtask queue is now empty.
6
Macrotask queue — pick one. The setTimeout callback runs: console.log("2"). Output: "2"

Final output order: 1 → 4 → 3 → 2

The Pattern to Memorize Synchronous code always runs first. Then microtasks (Promises) run in full. Then one macrotask (setTimeout). Then microtasks again. Repeat. The order is always: sync → microtasks → one macrotask → microtasks → one macrotask → ...

A Harder Variation

console.log("start");

setTimeout(() => {
  console.log("timeout");
  Promise.resolve().then(() => console.log("promise inside timeout"));
}, 0);

Promise.resolve().then(() => {
  console.log("promise 1");
}).then(() => {
  console.log("promise 2");
});

console.log("end");

// Output:
// "start"
// "end"
// "promise 1"
// "promise 2"
// "timeout"
// "promise inside timeout"

Notice: "promise inside timeout" logs after "timeout" but before any other macrotask would run. Why? Because after the "timeout" macrotask runs, all microtasks are drained again — and "promise inside timeout" is now in the microtask queue.
