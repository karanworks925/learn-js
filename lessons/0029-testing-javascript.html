Testing JavaScript

Untested code is unfinished code — seniors write tests that make them move faster, not slower.

Primary Source for this lesson Read this after the lesson: Jest — Getting Started. Jest is the most widely used JavaScript testing framework. This is the canonical reference.

1. Why Test?

Many developers skip tests and think it saves time. It doesn't — it borrows time from your future self at an extremely high interest rate.

Without TestsWith Tests
You're afraid to refactorYou refactor fearlessly
Every change might break something invisibleRegressions are caught in seconds
Bugs find your users firstBugs find your terminal first
Code is a black box even to youTests document how the code is meant to be used
Bugs in production cost 10-100x more to fixBugs caught at test time cost almost nothing
Tests are a Multiplier A developer with good test coverage moves slower at first but accelerates over time. A developer without tests moves fast at first and grinds to a halt as complexity grows. Senior developers know this and write tests by default.

2. Types of Tests

TypeWhat It TestsSpeedQuantity
UnitA single function or module in isolationMillisecondsMany — hundreds or thousands
IntegrationHow multiple units work together (e.g., a service + database)SecondsSome — dozens
End-to-End (E2E)The entire app in a real browser (e.g., Playwright, Cypress)MinutesFew — key user flows only

The testing pyramid: invest most in unit tests (fast, cheap, focused), fewer integration tests, and only a small number of E2E tests for critical paths. Inverting the pyramid — relying on E2E tests for everything — makes your test suite slow, brittle, and expensive.

3. The AAA Pattern

Every well-written test follows the same structure: Arrange, Act, Assert. When you see a test that's hard to understand, it's usually because these three phases are tangled together.

// calculator.js
function divide(a, b) {
  if (b === 0) throw new Error("Division by zero");
  return a / b;
}

// calculator.test.js
describe("divide", () => {
  it("divides two numbers", () => {
    // ARRANGE: set up the data
    const a = 10;
    const b = 2;

    // ACT: run the code under test
    const result = divide(a, b);

    // ASSERT: verify the result
    expect(result).toBe(5);
  });

  it("throws when dividing by zero", () => {
    // ARRANGE
    const divide_by_zero = () => divide(10, 0);

    // ACT + ASSERT
    expect(divide_by_zero).toThrow("Division by zero");
  });
});

4. Writing Pure Functions Makes Testing Easy

A pure function has no side effects and always returns the same output for the same input. Testing pure functions requires zero setup — no mocks, no database, no DOM, no network.

// IMPURE — hard to test (depends on external state)
function getGreeting(userId) {
  const user = db.getUser(userId);       // external dependency
  const hour = new Date().getHours();    // non-deterministic
  return hour < 12 ? `Good morning, ${user.name}` : `Hello, ${user.name}`;
}

// PURE — trivial to test
function getGreeting(name, hour) {
  return hour < 12 ? `Good morning, ${name}` : `Hello, ${name}`;
}

// Test needs zero setup:
expect(getGreeting("Karan", 9)).toBe("Good morning, Karan");
expect(getGreeting("Karan", 14)).toBe("Hello, Karan");
Design Insight If a function is hard to test, that is a signal it is poorly designed — not that testing is hard. When you push dependencies (like databases, clocks, and network calls) to the edges of your system and keep business logic pure, everything becomes easier: testing, reasoning, and changing the code.

5. Jest Basics

Jest is the standard JavaScript testing framework. It is built into Create React App, Next.js, and most modern JS toolchains. You need to know its vocabulary cold.

// calculator.js — the module we're testing
export function add(a, b) { return a + b; }
export function multiply(a, b) { return a * b; }
export function subtract(a, b) { return a - b; }

// calculator.test.js
import { add, multiply, subtract } from "./calculator";

describe("Calculator", () => {

  describe("add", () => {
    it("adds two positive numbers", () => {
      expect(add(2, 3)).toBe(5);       // strict equality (Object.is)
    });

    it("handles negative numbers", () => {
      expect(add(-1, 1)).toBe(0);
    });
  });

  it("multiplies correctly", () => {
    expect(multiply(3, 4)).toBe(12);
  });
});

Key Jest Matchers

MatcherChecksUse When
.toBe(val)Strict equality (Object.is)Primitives: numbers, strings, booleans
.toEqual(val)Deep equality (recursive)Objects and arrays
.toThrow(msg?)Function throwsError handling paths
.toBeTruthy()Value is truthyFlexible truthiness checks
.toBeFalsy()Value is falsyFlexible falsiness checks
.toContain(item)Array contains itemTesting array membership
.toHaveBeenCalled()Mock was calledVerifying side effects
.toHaveBeenCalledWith(...args)Mock called with specific argsVerifying call arguments
toBe vs toEqual — The Critical Difference expect({a:1}).toBe({a:1}) FAILS — two different objects, different references. expect({a:1}).toEqual({a:1}) PASSES — same structure. Use toBe for primitives, toEqual for objects and arrays.

6. Mocking

Mocking replaces real dependencies with controlled fakes. You mock things that are slow (network, DB), non-deterministic (Date.now), or have side effects (sending emails, writing files).

// userService.js
import { fetchUser } from "./api";

export async function getUserName(id) {
  const user = await fetchUser(id);
  return user.name.toUpperCase();
}

// userService.test.js
import { getUserName } from "./userService";
import { fetchUser } from "./api";

jest.mock("./api"); // replaces the entire module with auto-mocks

it("returns uppercased user name", async () => {
  // Control what the mock returns for this test
  fetchUser.mockResolvedValue({ name: "karan", id: 1 });

  const result = await getUserName(1);

  expect(result).toBe("KARAN");
  expect(fetchUser).toHaveBeenCalledWith(1); // verify it was called correctly
});

Mock Functions (jest.fn)

// Create a mock function from scratch
const mockCallback = jest.fn();

mockCallback("hello");
mockCallback("world");

expect(mockCallback).toHaveBeenCalledTimes(2);
expect(mockCallback).toHaveBeenCalledWith("hello");
expect(mockCallback.mock.calls[0][0]).toBe("hello"); // first call, first arg

// Make it return a value:
const mockAdd = jest.fn((a, b) => a + b);
expect(mockAdd(2, 3)).toBe(5);

Controlling Timers

it("debounce only fires once after delay", () => {
  jest.useFakeTimers(); // take control of setTimeout/setInterval

  const fn = jest.fn();
  const debounced = debounce(fn, 300);

  debounced(); debounced(); debounced();

  expect(fn).not.toHaveBeenCalled(); // not yet

  jest.runAllTimers(); // fast-forward all timers

  expect(fn).toHaveBeenCalledTimes(1); // only once

  jest.useRealTimers(); // restore real timers
});

7. TDD — Test-Driven Development

TDD flips the normal workflow: you write the test before you write the code. This sounds strange at first, but it produces better-designed, better-tested code by forcing you to think about the interface before the implementation.

The Red → Green → Refactor Cycle

  1. Red — Write a test for the behavior you want. Run it. It fails (red) because the code doesn't exist yet.
  2. Green — Write the minimum code needed to make the test pass. Don't over-engineer. Just make it green.
  3. Refactor — Clean up the code: remove duplication, improve naming, simplify logic. The tests keep you safe.
// STEP 1 — RED: write the test first
it("formats a price with currency symbol", () => {
  expect(formatPrice(9.99, "USD")).toBe("$9.99");
  expect(formatPrice(10, "EUR")).toBe("€10.00");
});
// Run: FAIL — formatPrice is not defined

// STEP 2 — GREEN: write minimum code to pass
const symbols = { USD: "$", EUR: "€", GBP: "£" };

function formatPrice(amount, currency) {
  return `${symbols[currency]}${amount.toFixed(2)}`;
}
// Run: PASS

// STEP 3 — REFACTOR: improve without breaking
function formatPrice(amount, currency) {
  return new Intl.NumberFormat("en", { style: "currency", currency })
    .format(amount);
}
// Run: still PASS — tests caught any regression
TDD Benefits When you write tests first, you naturally design your code to be testable (which means: modular, with clear interfaces). You get 100% coverage of the features you build for free. And you always know exactly when you're done — when the tests pass.
