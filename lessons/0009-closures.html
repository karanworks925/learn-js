Closures

The most important JavaScript concept — and the most asked about in interviews.

📖 Primary Source for this lesson Read this after the lesson: javascript.info — Closures. This is one of the most thorough explanations of closures on the internet. Re-read it until every section clicks.

1. What is a Closure?

A closure is a function that remembers the variables from its outer lexical environment even after that outer function has finished executing and returned.

Simpler: a closure is a function bundled together with the surrounding state (variables) it was created in.

Here is the classic example — a counter that persists its state between calls:

function makeCounter() {
  let count = 0; // this variable lives in makeCounter's Lexical Environment

  return function() {
    count++;           // this inner function closes over count
    return count;
  };
}

const counter = makeCounter(); // makeCounter runs and RETURNS the inner function
                                  // makeCounter is done — but its Lexical Environment lives on

console.log(counter()); // 1
console.log(counter()); // 2
console.log(counter()); // 3

// Each counter is independent
const counter2 = makeCounter();
console.log(counter2()); // 1 — fresh count, completely separate
console.log(counter());  // 4 — the first counter keeps going
The key insight count is not inside the returned function. It is in the Lexical Environment of makeCounter. The returned function holds a live reference to that environment — so every time you call counter(), it reads and updates the same count variable.

2. How Closures Work Under the Hood

When a function is created (not called), JavaScript attaches a hidden internal slot called [[Environment]] to it. This slot holds a reference to the Lexical Environment that was active at the moment the function was created.

Step-by-step walkthrough of the counter example:

  1. makeCounter() is called. A new Lexical Environment is created: { count: 0 }. Its outer reference points to the global LE.
  2. The inner function is created inside makeCounter. Its [[Environment]] is set to makeCounter's LE: { count: 0 }.
  3. makeCounter returns the inner function and finishes executing.
  4. Normally, makeCounter's LE would be garbage collected. But the returned inner function still references it — so it stays in memory.
  5. When you call counter(), it runs inside its [[Environment]]. It finds count there, increments it, and returns it.
  6. The LE now holds { count: 1 }. It persists for the next call.
// Memory model (simplified):
//
// counter function object
//   [[Environment]] ──► { count: 3 }  ──► global LE
//                         ▲
//                         This LE persists as long as counter exists

3. Practical Use 1: Data Encapsulation / Private State

Closures let you create variables that are truly private — inaccessible from outside the function. This is how you write encapsulated modules in plain JavaScript without classes.

function createBankAccount(initialBalance) {
  let balance = initialBalance; // PRIVATE — nothing outside can touch this directly

  return {
    deposit(amount) {
      if (amount > 0) balance += amount;
      return balance;
    },
    withdraw(amount) {
      if (amount > balance) throw new Error("Insufficient funds");
      balance -= amount;
      return balance;
    },
    getBalance() {
      return balance; // controlled read access
    }
  };
}

const account = createBankAccount(100);
console.log(account.deposit(50));   // 150
console.log(account.withdraw(30)); // 120
console.log(account.balance);       // undefined — direct access is blocked

4. Practical Use 2: Factory Functions

A factory function is a function that returns a customized function. The returned function closes over the factory's parameters, "remembering" its configuration forever.

// Multiplier factory
const multiplier = (n) => (x) => x * n;

const double = multiplier(2);   // double closes over n=2
const triple = multiplier(3);   // triple closes over n=3
const times10 = multiplier(10); // times10 closes over n=10

console.log(double(5));   // 10
console.log(triple(5));   // 15
console.log(times10(5)); // 50

// Real-world example: URL builder
const apiEndpoint = (base) => (path) => `${base}${path}`;

const devApi = apiEndpoint("http://localhost:3000");
const prodApi = apiEndpoint("https://api.myapp.com");

console.log(devApi("/users"));   // "http://localhost:3000/users"
console.log(prodApi("/users"));  // "https://api.myapp.com/users"

5. Practical Use 3: Event Handlers with Remembered State

Closures are what make event handlers stateful — each handler can remember data from when it was created.

// A button that counts how many times it has been clicked
function makeClickCounter(buttonLabel) {
  let clicks = 0;

  return function handleClick() {
    clicks++;
    console.log(`${buttonLabel} clicked ${clicks} time(s)`);
  };
}

const onSaveClick = makeClickCounter("Save");
const onDeleteClick = makeClickCounter("Delete");

// In a real app: button.addEventListener("click", onSaveClick);
onSaveClick();   // "Save clicked 1 time(s)"
onSaveClick();   // "Save clicked 2 time(s)"
onDeleteClick(); // "Delete clicked 1 time(s)" — independent count

6. The Classic var-in-Loop Bug (Interview Favorite)

This is one of the most commonly asked JavaScript interview questions. Read it carefully.

// THE BUG
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}
// Prints: 3, 3, 3 — NOT 0, 1, 2

Why? Because var is function-scoped, there is only ONE i variable shared by all three callbacks. By the time the setTimeout callbacks fire (after the loop finishes), i is already 3. All three callbacks close over the same i.

Fix 1: Use let (the modern solution)

for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}
// Prints: 0, 1, 2 — correct!
// let creates a NEW binding for i in each iteration's block scope.
// Each callback closes over a DIFFERENT i.

Fix 2: IIFE (Immediately Invoked Function Expression)

The historical fix before let existed. Creates a new scope per iteration by immediately invoking a function with the current value of i.

for (var i = 0; i < 3; i++) {
  (function(capturedI) {
    setTimeout(() => console.log(capturedI), 0);
  })(i); // Immediately call, passing current i as argument
}
// Prints: 0, 1, 2 — correct!
// Each IIFE invocation gets its own capturedI parameter (0, 1, 2).
// The callback closes over capturedI, not i.
In interviews When asked about the loop bug, mention all three options: the bug (var, shared scope), Fix 1 (let, per-iteration block scope), and Fix 2 (IIFE, pre-let historical solution). Showing you know all three signals deep understanding.

7. Interview: How to Answer "What is a Closure?"

This question appears in almost every JavaScript interview. Here is a model answer you should memorize and make your own:

Model answer "A closure is a function that retains access to its outer lexical environment — the variables and parameters of the function it was defined inside — even after that outer function has returned. In JavaScript, every function creates a closure when it is defined. The inner function gets a hidden [[Environment]] reference that keeps the outer scope alive in memory as long as the inner function exists. This enables patterns like private state, factory functions, and memoization."

Common follow-up questions and how to handle them:

Follow-up QuestionKey Point in Your Answer
"Give me a real-world use case"Private state (counter, bank account), factory functions (multiplier, API builders), event handlers, memoization, partial application
"Does a closure cause memory leaks?"Yes, if you hold a closure that references large outer objects and never release it. The outer LE won't be GC'd. Solution: nullify references when done.
"What's the var-in-loop bug?"All callbacks close over the same var. Fix: use let (new binding per iteration) or IIFE (capture value in new function scope).
"How is a closure different from a class?"Both can encapsulate state. Closures are simpler, function-based, have no prototype chain overhead. Classes are more structured and better for multiple instances with shared methods.
