Functions

Functions are the primary building blocks of JavaScript programs.

Primary Sources for this lesson Read these after the lesson to go deeper: javascript.info — Function Basics & javascript.info — Arrow Functions. These cover everything in this lesson with additional examples.

1. Function Declarations

A function declaration uses the function keyword followed by a name. This is the most familiar syntax and the one that gets fully hoisted.

// Function declaration — can be called BEFORE it appears in the file
greet("Karan"); // ✅ Works! Declarations are fully hoisted

function greet(name) {
  console.log(`Hello, ${name}!`);
}

// Naming convention: camelCase, verb-noun pattern
function getUserById(id) { /* ... */ }
function calculateTax(amount, rate) { /* ... */ }
function isValidEmail(email) { /* ... */ } // boolean functions: is/has/can prefix
Hoisting Recap Function declarations are hoisted completely — name and body. JS processes all declarations before running any code. This is why you can call a declared function before the line it appears on.

2. Function Expressions

A function expression assigns a function to a variable. The variable is hoisted (as undefined for var, or into the TDZ for let/const), but the function itself is not.

// Function expression — cannot be called before its line
sayHi(); // ❌ TypeError: sayHi is not a function (var) or ReferenceError (const)

const sayHi = function() {
  console.log("Hi!");
};

// Named function expression — useful for recursion and stack traces
const factorial = function fact(n) {
  if (n <= 1) return 1;
  return n * fact(n - 1); // can reference itself by name
};
console.log(factorial(5)); // 120

3. Arrow Functions

Arrow functions (ES6) are a concise syntax that also fundamentally differ from regular functions in two ways: they have no own this and no arguments object. These differences matter a lot — you'll see why in the closures and this lessons.

// Various arrow function forms
const double = x => x * 2;           // one param: no parens needed; body is returned
const add = (a, b) => a + b;          // two params: parens required
const greet = () => "Hello!";         // no params: empty parens required
const square = x => ({ value: x * x }); // returning an object: wrap in parens

// Multi-line body — needs explicit return
const processUser = (user) => {
  const cleaned = user.name.trim();
  return { ...user, name: cleaned }; // explicit return required with braces
};

// Arrow functions shine as callbacks
const numbers = [3, 1, 4, 1, 5];
const doubled = numbers.map(n => n * 2);       // [6, 2, 8, 2, 10]
const evens   = numbers.filter(n => n % 2 === 0); // [4]
Arrow Functions Cannot: Be used with new (no constructor), access arguments object, or have their own this binding. Trying new (() => {})() throws a TypeError. For object methods and constructors, use regular functions.

4. Parameters & Arguments

Understanding how JS handles parameters is critical for avoiding silent bugs.

// Missing arguments become undefined
function greet(name, greeting) {
  console.log(greeting, name);
}
greet("Karan"); // undefined "Karan" — greeting is undefined

// Default parameter values (ES6)
function greet(name, greeting = "Hello") {
  console.log(greeting, name);
}
greet("Karan");           // "Hello Karan"
greet("Karan", "Hey");    // "Hey Karan"
greet("Karan", undefined); // "Hello Karan" — undefined triggers the default
// Rest parameters — collect remaining args into an array
function sum(...nums) {            // nums is a real Array
  return nums.reduce((total, n) => total + n, 0);
}
sum(1, 2, 3, 4); // 10

// Mix of named params and rest
function logEvent(type, ...details) {
  console.log(`[${type}]`, ...details);
}
logEvent("ERROR", "File not found", "/path/to/file");
// [ERROR] File not found /path/to/file

// The old way: arguments object (avoid in new code)
function oldSum() {
  return Array.from(arguments).reduce((a, b) => a + b, 0);
  // arguments is array-LIKE, not a real array — has no .reduce() directly
}
Rest vs Arguments Rest parameters (...args) give you a real Array — you can call .map(), .filter(), etc. directly. The old arguments object is array-like but not an array. Always prefer rest parameters in new code.

5. Return Values

// Explicit return
function add(a, b) {
  return a + b; // stops execution and returns this value
  console.log("never reached"); // unreachable code
}

// No return statement → returns undefined
function logThings() {
  console.log("hello");
  // implicit: return undefined
}
const result = logThings(); // result is undefined

// Arrow function implicit return (no braces = auto-return)
const triple = n => n * 3;         // implicitly returns n * 3
const noReturn = n => { n * 3; };  // braces → no implicit return → undefined!

// Early return keeps code clean
function divide(a, b) {
  if (b === 0) return null; // guard clause
  return a / b;
}

6. First-Class Functions

In JavaScript, functions are values — just like numbers and strings. This single fact unlocks callbacks, higher-order functions, and closures. It is what makes JS a functional-capable language.

// Functions can be stored in variables
const sayHello = function() { return "hello"; };

// Functions can be stored in objects
const utils = {
  double: x => x * 2,
  square: x => x ** 2,
};

// Functions can be passed as arguments (callbacks)
function runTwice(fn) {
  fn();
  fn();
}
runTwice(() => console.log("ping")); // logs "ping" twice

// Functions can be returned from other functions (higher-order functions)
function makeMultiplier(factor) {
  return (n) => n * factor; // returns a NEW function
}
const triple = makeMultiplier(3);
const quadruple = makeMultiplier(4);
console.log(triple(5));    // 15
console.log(quadruple(5)); // 20
Why This Matters First-class functions are the foundation of: callbacks (setTimeout, event listeners), array methods (map, filter, reduce), closures, and React patterns like custom hooks. Understand this deeply.

7. IIFE — Immediately Invoked Function Expression

An IIFE is a function that defines itself and runs immediately. Its primary use is creating a local scope to avoid polluting the global namespace.

// Basic IIFE syntax — wrap in parens, then invoke with ()
(function() {
  const secret = "I only live in here";
  console.log("IIFE ran");
})();

// Arrow function IIFE
(() => {
  console.log("Arrow IIFE");
})();

// IIFE with return value — capture the result
const appConfig = (function() {
  const env = process.env.NODE_ENV || "development";
  return {
    isDev: env === "development",
    apiUrl: env === "production" ? "https://api.prod.com" : "http://localhost:3000",
  };
})(); // appConfig is now the returned object

// Modern alternative: block scope with let/const does the same job
{
  const secret = "I only live in this block";
  console.log(secret);
}
// console.log(secret); ← ReferenceError — block scoped
When Do You See IIFEs Today? Less common in modern code since ES6 modules and block-scoped let/const solve the same scoping problem. But you'll see IIFEs in older codebases, browser scripts without modules, and some config patterns. Recognize them in interviews.
