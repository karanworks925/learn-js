Modules

Modules let you split code into files and compose them — the foundation of every real project.

Primary Source for this lesson Read this after the lesson to solidify what you learn: javascript.info — Modules Introduction. This is the highest-quality JS learning resource available.

1. Why Modules?

Before modules existed, all JavaScript lived in a single file or was loaded via multiple <script> tags that all shared the same global scope. This caused real problems in large codebases.

  • Global scope pollution — every variable was shared; any script could overwrite anything
  • No encapsulation — no way to make something private
  • Load order dependency — scripts had to be included in exactly the right order

The IIFE Workaround

Before modules, developers used Immediately Invoked Function Expressions to create a private scope and expose only what was needed:

const Utils = (function() {
  const _private = "hidden"; // not accessible outside

  function greet(name) {
    return "Hello, " + name;
  }

  return { greet }; // only expose what's needed
})();

Utils.greet("Karan"); // "Hello, Karan"

CommonJS — Node.js Modules

Node.js introduced CommonJS (CJS) before the language had a module system. It uses require() to import and module.exports to export. It is still widely used in Node.js environments today.

// math.js — CommonJS export
function add(a, b) { return a + b; }
module.exports = { add };

// main.js — CommonJS import
const { add } = require('./math');
console.log(add(2, 3)); // 5

ES Modules — The Standard Today

ES Modules (ESM) were standardized in ES2015 (ES6). They use import and export syntax and are now the official module system for both browsers and modern Node.js.

The Modern Default In any new project today — React, Vue, Svelte, modern Node.js — you will use ES Modules. Understanding both ESM and CommonJS is required for senior-level work because you will encounter both.

2. Named Exports

Named exports let you export multiple values from a single module. The name used when exporting must match the name used when importing (unless you rename it).

// utils.js — three ways to write named exports

// Method 1: export inline
export const name = "Karan";
export function greet(who) {
  return `Hello, ${who}!`;
}

// Method 2: export list at the end of the file (equally valid)
const version = "1.0.0";
function farewell(who) { return `Bye, ${who}!`; }
export { version, farewell };
// main.js — importing named exports

// Basic named import
import { name, greet } from './utils.js';
console.log(greet(name)); // "Hello, Karan!"

// Renaming on import — useful to avoid name collisions
import { greet as hello } from './utils.js';
hello("World"); // "Hello, World!"

// Namespace import — get everything as one object
import * as utils from './utils.js';
utils.greet("Karan"); // "Hello, Karan!"
console.log(utils.name); // "Karan"
Tip Prefer specific named imports (import { greet }) over namespace imports (import * as utils) — bundlers like Webpack and Vite can tree-shake (remove unused code) with named imports but not always with namespace imports.

3. Default Export

A module can have exactly one default export. It represents the main thing the module provides. When importing, you can use any name you want.

// Button.js — default export
export default function Button({ label, onClick }) {
  // component logic...
}

// You can also export a class, object, or any value as default:
export default { apiUrl: "https://api.example.com" };
// main.js — importing the default export
import Button from './Button.js';    // named "Button" by convention
import Btn    from './Button.js';    // same thing, different local name

// Mix default and named imports in one statement:
import App, { version, greet } from './app.js';

Default vs Named — When to Use Each

SituationUse
A file exports one main thing (a component, a class, a config object)Default export
A file exports multiple utilities or functionsNamed exports
You want to enforce a consistent name across the codebaseNamed export (name is enforced at import)
You want flexible naming at the import siteDefault export (consumer picks the name)
Style Warning Many style guides (including Airbnb's) discourage anonymous default exports like export default function() {} because they appear as "anonymous" in stack traces. Always name your exported functions even when using default export.

4. Re-exporting & Barrel Exports

As projects grow, you often want a single entry point that re-exports from many files. This is called a barrel export and is typically done in an index.js file.

// components/Button.js
export default function Button() {}

// components/Input.js
export function Input() {}
export function Textarea() {}

// components/index.js — the barrel file
export { default } from './Button.js';      // re-export default
export { default as Button } from './Button.js'; // re-export default as named
export { Input, Textarea } from './Input.js';  // re-export named
// Now consumers import from ONE place instead of many:
import { Button, Input, Textarea } from './components';
// Instead of:
import Button   from './components/Button.js';
import { Input } from './components/Input.js';
Barrel Files in Practice Every popular UI library (React, Material UI, Chakra UI) uses barrel exports. When you do import {'{'} useState, useEffect {'}'} from 'react', you are using React's barrel export from its main index file.

5. CommonJS vs ESM — Side by Side

FeatureCommonJS (CJS)ES Modules (ESM)
Import syntaxrequire('./module')import x from './module.js'
Export syntaxmodule.exports = ...export ...
LoadingSynchronousAsynchronous
AnalysisDynamic (run-time)Static (parse-time)
Tree shakingNot possibleSupported by bundlers
Top-level awaitNot supportedSupported
Browser supportNo (needs bundler)Yes (<script type="module">)
Node.js supportDefault (.js files).mjs or "type":"module"

The key architectural difference is that ESM imports are static — they must appear at the top level of a file, and the engine knows the full import/export graph before any code runs. This is what enables tree-shaking.

Node.js Interop To use ESM in Node.js: either name your file .mjs, or add "type": "module" to your package.json. You cannot use import in a regular .js file without this configuration.

6. Dynamic import()

Sometimes you don't want to load a module at startup — you want to load it on demand. Dynamic import() returns a Promise and can appear anywhere in your code, not just at the top level.

// Static import — always loaded at startup
import { heavyChart } from './charts.js';

// Dynamic import — loaded only when needed
async function showChart() {
  // This only loads charts.js when the user clicks "Show Chart"
  const { heavyChart } = await import('./charts.js');
  heavyChart();
}

// Dynamic import works with expressions too (unlike static import)
const locale = "fr";
const messages = await import(`./locales/${locale}.js`);

Dynamic import() is the basis of code splitting in React (via React.lazy()) and all modern bundlers. Instead of sending one giant JavaScript bundle, the app loads only what the user needs right now.

What import() Returns It returns a Promise that resolves to a module namespace object — equivalent to what you get from import * as module. So if your module has a default export, access it as module.default.

7. Module Scope

Every ES Module runs in its own scope. Top-level variables are local to that module — they are not added to window or the global object.

// module-a.js
const secret = "only mine";
console.log(this); // undefined in strict mode (modules are always strict)

// module-b.js
console.log(secret); // ❌ ReferenceError — not accessible
console.log(typeof secret); // "undefined"
Modules Are Always Strict ES Modules run in strict mode automatically — you do not need to write "use strict". This means: no undeclared variables, no duplicate parameter names, this is undefined at the top level (not window).

Modules are also singletons — if two files import the same module, they both get the same instance. Exports are live bindings: if a module changes an exported value, all importers see the updated value.

// counter.js
export let count = 0;
export function increment() { count++; }

// a.js
import { count, increment } from './counter.js';
increment();
console.log(count); // 1 — live binding reflects the change
