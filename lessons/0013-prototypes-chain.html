Prototypes & the Prototype Chain

JavaScript's inheritance model — everything classes are built on.

Primary Source for this lesson Read this after the lesson: javascript.info — Prototype Inheritance. This is one of the most important chapters in all of JS. Re-read it at least twice.

1. Every Object Has a Prototype

In JavaScript, every object has a hidden internal slot called [[Prototype]]. This slot holds a reference to another object — or null at the very end of the chain. This linked object is called the object's prototype.

The prototype itself is also just an object, so it too has a [[Prototype]]. This creates a chain. At the top of this chain for all plain objects sits Object.prototype, whose own [[Prototype]] is null.

// Visualizing the chain for a plain object literal:
//
//  myObj  →  Object.prototype  →  null
//
// myObj has its own properties
// Object.prototype has: toString, hasOwnProperty, valueOf, etc.
// null = end of the chain

const myObj = { name: "Karan" };

// You can verify the chain:
Object.getPrototypeOf(myObj) === Object.prototype; // true
Object.getPrototypeOf(Object.prototype);          // null
Why does this matter? This is how methods like .toString(), .hasOwnProperty(), and .valueOf() are available on every object you create. You never defined them — they live on Object.prototype and your object inherits them through the chain.

2. Accessing the Prototype

const obj = { x: 1 };

// CORRECT — official ES5+ method
Object.getPrototypeOf(obj); // returns Object.prototype

// DEPRECATED — still works but avoid in production
obj.__proto__; // same result, but __proto__ is non-standard/deprecated

// Setting a prototype at creation time
const animal = { breathes: true };
const dog = Object.create(animal);
Object.getPrototypeOf(dog) === animal; // true
__proto__ is deprecated You will see __proto__ everywhere in older code and Stack Overflow answers. Understand it, but use Object.getPrototypeOf() in anything you write.

3. Property Lookup via the Prototype Chain

When you write obj.prop, JavaScript doesn't just look at obj. It follows a specific algorithm:

  1. Look for prop as an own property of obj. If found, return it.
  2. If not found, move to obj's [[Prototype]] and repeat.
  3. Keep walking up the chain.
  4. If null is reached (end of chain) without finding the property, return undefined.
const animal = {
  breathes: true,
  describe() {
    return `I breathe: ${this.breathes}`;
  },
};

const dog = Object.create(animal);
dog.name = "Rex"; // own property on dog

// Lookup: dog.name → found on dog itself
dog.name;      // "Rex"

// Lookup: dog.breathes → NOT on dog, found on animal (dog's prototype)
dog.breathes;  // true

// Lookup: dog.describe() → NOT on dog, found on animal
dog.describe();  // "I breathe: true" — 'this' refers to dog

// Lookup: dog.toString() → not on dog, not on animal
//         found on Object.prototype (the end of the chain)
dog.toString(); // "[object Object]"

// Not found at all
dog.fly;  // undefined — reached null, not found
Mental model Think of the prototype chain as a series of lookup tables. When you ask for a property, JS checks each table in order, starting with the object itself. The first match wins. If no table has the property, you get undefined.

4. Object.create()

Object.create(proto) creates a brand-new empty object whose [[Prototype]] is set to proto. It's the cleanest, most explicit way to set up a prototype chain.

const vehicleProto = {
  start() { return `${this.make} started`; },
  stop()  { return `${this.make} stopped`; },
};

const car = Object.create(vehicleProto);
car.make = "Toyota";
car.model = "Corolla";

car.start(); // "Toyota started" — method found on prototype

// Object.create(null) — no prototype at all
// Creates a pure dictionary with no inherited methods
const dict = Object.create(null);
dict.name = "Karan";
// dict.toString is undefined — no inherited methods
// Useful when you want a clean map with no prototype interference

5. hasOwnProperty and the in Operator

These two are frequently confused in interviews. The difference is whether they look only at own properties or traverse the chain.

const animal = { breathes: true };
const dog = Object.create(animal);
dog.name = "Rex";

// hasOwnProperty — only own properties, NOT inherited
dog.hasOwnProperty("name");     // true  — own property
dog.hasOwnProperty("breathes"); // false — inherited, not own

// 'in' operator — checks own AND prototype chain
"name" in dog;     // true
"breathes" in dog; // true  — found via prototype chain
"fly" in dog;      // false — not found anywhere

// Modern alternative (ES2022) — safer than hasOwnProperty
Object.hasOwn(dog, "name");     // true
Object.hasOwn(dog, "breathes"); // false

6. Constructor Functions — The Pre-Class Pattern

Before ES6 classes, JavaScript developers used constructor functions to create many objects sharing the same prototype. You still need to understand this pattern because you'll encounter it in older codebases and interview questions.

// Convention: constructor functions start with a capital letter
function Person(name, age) {
  this.name = name;
  this.age = age;
}

// Methods go on the prototype — shared by ALL instances
Person.prototype.greet = function() {
  return `Hi, I'm ${this.name}`;
};

const alice = new Person("Alice", 28);
const bob   = new Person("Bob", 34);

alice.greet(); // "Hi, I'm Alice"
bob.greet();   // "Hi, I'm Bob"

// Both instances share the SAME greet function via the prototype
alice.greet === bob.greet; // true — one function, many instances

What new does — step by step

When you write new Person("Alice", 28), JavaScript does four things automatically:

  1. Creates a new empty object {}
  2. Sets that object's [[Prototype]] to Person.prototype
  3. Runs the constructor function with this pointing to the new object
  4. Returns the new object (unless the constructor explicitly returns a different object)
Why put methods on the prototype? If you put greet inside the constructor as this.greet = function() {...}, a new function is created for every instance. With 1000 Person objects, that's 1000 identical functions in memory. Putting it on Person.prototype means all instances share one function. Always put methods on the prototype.

7. The prototype Property vs [[Prototype]]

This is the most commonly confused concept about prototypes — even experienced developers mix it up. There are two different things:

ThingWhat it isWho has it
Person.prototypeAn ordinary object that will become the [[Prototype]] of instances created by new Person()Functions (specifically, constructor functions)
obj.[[Prototype]]The actual prototype link of an object — accessed via Object.getPrototypeOf(obj)Every object
 
function Dog(name) {
  this.name = name;
}
Dog.prototype.bark = function() { return "Woof!"; };

const rex = new Dog("Rex");

// rex's [[Prototype]] IS Dog.prototype
Object.getPrototypeOf(rex) === Dog.prototype; // true
rex.__proto__ === Dog.prototype;               // true (deprecated but shown for clarity)

// Dog.prototype's [[Prototype]] is Object.prototype
Object.getPrototypeOf(Dog.prototype) === Object.prototype; // true

// Full chain: rex → Dog.prototype → Object.prototype → null
Interview trap When an interviewer asks "what is the prototype of rex?" the answer is Dog.prototype. Dog itself is the constructor function — not the prototype. The prototype is the object that Dog.prototype points to.
