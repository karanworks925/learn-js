Functional Programming

Write code that's predictable, testable, and composable — the functional way.

Primary Source for this lesson Read this after the lesson: Eloquent JavaScript — Higher-Order Functions (Chapter 5). It's one of the best introductions to functional thinking in JavaScript.

1. What Is Functional Programming?

Functional Programming (FP) is a programming paradigm built around the idea of composing pure functions and avoiding shared mutable state. It's not about using .map() and .filter() — those are just tools. FP is a way of thinking about your program.

ParadigmCore ideaJS example
ImperativeTell the computer HOW to do it, step by stepfor loops, mutation in place
Object-OrientedGroup state and behavior into objectsClasses, this, encapsulation
FunctionalTransform data through pure functions.map(), composition, immutability

JavaScript is multi-paradigm — you can mix all three. But FP shines for data transformation pipelines, and it's the mental model behind React's design.

Key FP principles:

  • Pure functions
  • Immutability — don't mutate data, create new copies
  • Function composition — build complex behavior from simple pieces
  • Declarative style — describe WHAT you want, not HOW to get it

2. Pure Functions

A function is pure if it satisfies two conditions:

  1. Deterministic — same inputs always produce the same output
  2. No side effects — it doesn't modify anything outside its own scope
// PURE — same input, same output, no external changes
function add(a, b) {
  return a + b;
}
add(2, 3); // always 5, every time, forever

// IMPURE — result changes based on external state
let tax = 0.2;
function priceWithTax(price) {
  return price * (1 + tax); // depends on external `tax` variable
}

// IMPURE — mutates its argument
function addItem(cart, item) {
  cart.push(item); // mutates the original array!
  return cart;
}

// PURE version — returns a new array
function addItem(cart, item) {
  return [...cart, item]; // original cart untouched
}

// These are always impure (by definition):
Math.random()  // different every call
Date.now()     // different every call
Why pure functions are amazing for your career Pure functions are trivial to unit test — call the function with an input, assert on the output. No mocks, no setup, no cleanup. Senior engineers prize testable code. If you write pure functions, you write testable code almost for free.

3. Side Effects — Necessary But Dangerous

A side effect is anything a function does beyond computing a return value:

  • Modifying a variable outside its scope
  • Writing to the DOM
  • Making a network request
  • Writing to localStorage, a database, or a file
  • Logging to the console
  • Mutating a function's arguments

Side effects are unavoidable — every useful program eventually has to interact with the outside world. The FP approach is to isolate them: keep the core of your program as pure functions, and push side effects to the outer edges.

// The pattern: pure core, impure shell

// PURE core — all the logic, no side effects
function filterActiveUsers(users) {
  return users.filter(u => u.active);
}
function formatUser(user) {
  return `${user.name} (${user.email})`;
}

// IMPURE shell — side effects happen here, at the boundary
async function displayActiveUsers() {
  const users = await fetch('/api/users').then(r => r.json()); // I/O
  const active = filterActiveUsers(users);      // pure
  const formatted = active.map(formatUser);      // pure
  document.querySelector('#list').textContent = formatted.join('\n'); // DOM
}

4. Immutability

Immutability means: never change data in place — create a new copy with the modification instead. This makes bugs far easier to track because you always know exactly what changed and when.

// MUTABLE approach — mutates the original
const user = { name: 'Karan', score: 10 };
user.score = 20; // original changed — who changed it? when?

// IMMUTABLE approach — create a new object
const updatedUser = { ...user, score: 20 }; // spread copies properties
// user is still { name: 'Karan', score: 10 }
// updatedUser is { name: 'Karan', score: 20 }

// Arrays — immutable equivalents of mutating methods
const items = [1, 2, 3];

// Instead of push → spread
const withFour = [...items, 4]; // [1, 2, 3, 4]

// Instead of splice (remove index 1) → filter
const withoutTwo = items.filter((_, i) => i !== 1); // [1, 3]

// Instead of sort in place → slice first to copy
const sorted = [...items].sort(); // items is untouched

// Object.freeze — shallow immutability enforcement
const config = Object.freeze({ apiUrl: 'https://api.example.com' });
config.apiUrl = 'hacked'; // silently fails in sloppy mode, throws in strict
console.log(config.apiUrl); // still 'https://api.example.com'
Object.freeze is shallow Object.freeze(obj) only prevents modifications to the top-level properties. Nested objects are still mutable. For deep immutability, use a library like Immer, or structure your data to avoid nesting mutations.

5. Function Composition

Composition means combining small, focused functions into larger ones. The output of one function becomes the input of the next. This keeps every function small, named, and testable.

// Three focused functions
const sanitize = str => str.trim().toLowerCase();
const normalize = str => str.replace(/\s+/g, '-');
const format = str => `slug: ${str}`;

// Manual composition — right to left, hard to read
const result = format(normalize(sanitize('  Hello World  ')));
// → 'slug: hello-world'

// compose — creates a right-to-left pipeline
const compose = (...fns) => x => fns.reduceRight((v, f) => f(v), x);
const makeSlug = compose(format, normalize, sanitize);
makeSlug('  Hello World  '); // → 'slug: hello-world'

// pipe — left-to-right (easier to read)
const pipe = (...fns) => x => fns.reduce((v, f) => f(v), x);
const makeSlug2 = pipe(sanitize, normalize, format);
makeSlug2('  Hello World  '); // → 'slug: hello-world'
// pipe reads in the order the transforms happen — most natural
pipe vs compose Both do composition. pipe applies functions left-to-right (read order = execution order), compose applies right-to-left (mathematical notation). pipe is usually easier to read and reason about. Pick one and be consistent.

6. Currying

Currying transforms a function that takes multiple arguments into a chain of single-argument functions. f(a, b) becomes f(a)(b).

This enables partial application — lock in some arguments now, supply the rest later. The result is a specialized, reusable function.

// Normal function
function multiply(a, b) {
  return a * b;
}

// Curried version
const multiply = a => b => a * b;

// Partial application — lock in the first argument
const double = multiply(2);   // returns b => 2 * b
const triple = multiply(3);   // returns b => 3 * b

double(5);  // 10
triple(5); // 15

// Practical: curried event handler
const handleInput = fieldName => event => {
  console.log(`${fieldName}:`, event.target.value);
};

document.querySelector('#email')
  .addEventListener('input', handleInput('email'));
document.querySelector('#name')
  .addEventListener('input', handleInput('name'));

// Practical: curried fetch with a base URL
const fetchFrom = baseUrl => path =>
  fetch(`${baseUrl}${path}`).then(r => r.json());

const apiGet = fetchFrom('https://api.example.com');
apiGet('/users');    // fetches /users
apiGet('/products'); // fetches /products

7. Point-Free Style

Point-free (also called tacit) style means writing functions without explicitly mentioning their arguments. Instead of wrapping a function, you pass it directly.

const numbers = [1, 2, 3, 4, 5];

// Not point-free — argument (n) is explicitly named
const doubled = numbers.map(n => n * 2);

// Point-free — pass the curried multiply(2) directly
const multiply = a => b => a * b;
const doubled2 = numbers.map(multiply(2));

// Another example — filtering
const users = [
  { name: 'Karan', active: true },
  { name: 'Raj', active: false }
];

// Not point-free
const active = users.filter(u => u.active);

// Point-free with a helper
const prop = key => obj => obj[key];
const active2 = users.filter(prop('active'));
Don't over-do it Point-free style can make pipelines cleaner and more readable. But taken too far, it becomes unreadable abstraction. The goal is clarity — use point-free when it helps, skip it when it obscures.
