Memory Management & Performance

Seniors write code that doesn't slow down or leak — understand how JavaScript manages memory.

Primary Source for this lesson Read this after the lesson: Chrome DevTools — Fix Memory Problems. Learn to use the Memory panel to catch leaks in real applications.

1. How JavaScript Manages Memory

Memory management in JavaScript happens in three phases. You don't control them manually, but understanding them explains why things go wrong.

  1. Allocation — Memory is reserved when you create variables, objects, arrays, or functions.
  2. Usage — Your program reads and writes that memory.
  3. Release — The garbage collector (GC) frees memory that is no longer reachable.

Stack vs Heap

StackHeap
Stores primitive values (number, string, boolean, etc.) and function call framesStores objects, arrays, functions, and closures
Fixed size, very fastDynamic size, slower
Automatically managed (LIFO)Managed by the garbage collector
The Practical Implication Primitive values are cheap — they live on the stack and disappear when the function returns. Objects live on the heap and stay there until no code references them. Memory leaks are almost always about objects that you think are gone, but still have a reference somewhere.

2. Garbage Collection

JavaScript's GC uses a mark-and-sweep algorithm. It works like this:

  1. Start from roots — the global object, the current call stack, closures in scope.
  2. Mark every object reachable from those roots (following all references).
  3. Sweep — delete everything that was NOT marked. That memory is freed.
let user = { name: "Karan" }; // object is reachable via 'user'
user = null;                   // no references remain → eligible for GC

let a = { data: "hello" };
let b = a;    // two references to the same object
a = null;    // still reachable via b — NOT collected
b = null;    // NOW it's collectable — zero references
V8 Generational GC V8 (Chrome/Node) separates objects into a "young generation" (recently created) and "old generation" (survived multiple collections). Short-lived objects are collected quickly and cheaply. Long-lived objects are compacted less frequently. This is why creating temporary objects in hot loops can cause GC pauses.

3. Memory Leaks — 4 Common Patterns

A memory leak happens when your code holds a reference to an object that it no longer needs. The GC can't collect it, so it sits in memory forever — growing with each page visit or operation.

Pattern 1: Accidental Global Variables

function processData() {
  result = "huge amount of data..."; // no let/const/var!
  // 'result' becomes a global variable — never collected
}

// Fix: always declare variables
function processData() {
  const result = "huge amount of data..."; // scoped, collected when done
}

Pattern 2: Forgotten Timers and Intervals

const hugeData = new Array(1000000).fill("data");

// LEAK: interval never cleared, hugeData is referenced forever
setInterval(() => {
  console.log(hugeData.length);
}, 1000);

// Fix: store the ID and clear it when done
const id = setInterval(() => {
  console.log(hugeData.length);
}, 1000);

// When the component/page unmounts:
clearInterval(id);

Pattern 3: Closures Retaining Large Scopes

function createHandler() {
  const HUGE_BUFFER = new Uint8Array(10_000_000); // 10MB

  return function handler() {
    // HUGE_BUFFER is captured in closure — even if unused here
    return "done";
  };
}

// Fix: don't close over large objects you don't need
function createHandler() {
  const HUGE_BUFFER = new Uint8Array(10_000_000);
  const result = processBuffer(HUGE_BUFFER); // extract what you need
  // HUGE_BUFFER can now be GC'd
  return () => result;
}

Pattern 4: Detached DOM Nodes

let detachedNode;

function setup() {
  const el = document.getElementById("myDiv");
  detachedNode = el; // stored in JS variable

  el.remove();      // removed from DOM
  // But detachedNode still holds a reference!
  // The entire subtree stays in memory
}

// Fix: null out references when removing elements
function teardown() {
  detachedNode.remove();
  detachedNode = null; // now it can be collected
}
Detecting Leaks in Chrome DevTools Open DevTools → Memory tab → take a Heap Snapshot. Look for "Detached HTMLElement" entries. Those are detached DOM node leaks. Use the Timeline allocation profiler to spot growing memory over time.

4. WeakMap and WeakSet

Regular Map and Set hold strong references. If you store an object as a key in a Map, that object will never be GC'd as long as the Map exists. WeakMap and WeakSet hold weak references — they don't prevent garbage collection.

Map / SetWeakMap / WeakSet
Reference typeStrong (prevents GC)Weak (allows GC)
Key typesAnythingObjects only (WeakMap keys, WeakSet values)
Iterable?Yes — forEach, for...ofNo — cannot be iterated
Has .size?YesNo
Use caseGeneral-purpose collectionsMetadata/caches tied to object lifetimes
 
// Attach metadata to DOM elements without preventing their collection
const elementData = new WeakMap();

function trackElement(el) {
  elementData.set(el, { clicks: 0, created: Date.now() });
}

function incrementClick(el) {
  const data = elementData.get(el);
  if (data) data.clicks++;
}

// When 'el' is removed from DOM and all JS refs drop,
// the WeakMap entry is automatically cleaned up — no manual removal needed

5. WeakRef and FinalizationRegistry (ES2021)

For advanced use cases, ES2021 added two more tools for working with GC.

// WeakRef: hold a weak reference to an object
let largeObject = { data: new Array(1000000) };
const ref = new WeakRef(largeObject);

largeObject = null; // may now be GC'd

// Later:
const obj = ref.deref(); // might be undefined if GC ran
if (obj) {
  console.log("Still alive");
} else {
  console.log("Collected");
}
// FinalizationRegistry: run a callback when an object is collected
const registry = new FinalizationRegistry((heldValue) => {
  console.log(`Object "${heldValue}" was garbage collected`);
});

let temp = { important: true };
registry.register(temp, "myTempObject");

temp = null; // Eventually: logs "Object 'myTempObject' was garbage collected"
Use With Care GC timing is not deterministic — you cannot predict when (or if) the callback fires. Never build critical application logic on FinalizationRegistry. It is useful only for cleanup and diagnostics, not for business logic.

6. Debounce

Debounce delays a function's execution until a specified time has passed since the last call. If the function is called again before the timer fires, the timer resets.

Use cases: search input (don't query the API on every keystroke), window resize handlers, form validation.

function debounce(fn, delay) {
  let timer;
  return function(...args) {
    clearTimeout(timer);
    timer = setTimeout(() => fn.apply(this, args), delay);
  };
}

// Usage:
const searchInput = document.getElementById("search");
const debouncedSearch = debounce(async (e) => {
  const results = await fetchResults(e.target.value);
  renderResults(results);
}, 300);

searchInput.addEventListener("input", debouncedSearch);
// API only called 300ms after typing stops

7. Throttle

Throttle limits a function to execute at most once per N milliseconds, regardless of how many times it is called. Unlike debounce, it fires on a regular interval during continuous calls.

Use cases: scroll events, mousemove handlers, rate-limited API calls.

function throttle(fn, limit) {
  let lastCall = 0;
  return function(...args) {
    const now = Date.now();
    if (now - lastCall >= limit) {
      lastCall = now;
      return fn.apply(this, args);
    }
  };
}

// Usage:
const throttledScroll = throttle(() => {
  updateScrollProgress();
}, 100);

window.addEventListener("scroll", throttledScroll);
// updateScrollProgress runs at most 10 times per second

Debounce vs Throttle — When to Use Which

DebounceThrottle
FiresOnce, after calls stopRegularly during continuous calls
Timer resetYes — each call resets the delayNo — fires on fixed intervals
Use caseSearch input, resize handlerScroll events, mousemove, API rate-limiting
Mental model"Wait until they stop""At most once per interval"

8. requestAnimationFrame

For animations, never use setInterval. Use requestAnimationFrame instead. The browser calls your callback just before the next repaint — roughly 60 times per second on 60Hz displays.

let position = 0;
let animationId;

function animate() {
  position += 2;
  element.style.transform = `translateX(${position}px)`;

  if (position < 500) {
    animationId = requestAnimationFrame(animate); // schedule next frame
  }
}

// Start:
animationId = requestAnimationFrame(animate);

// Stop:
cancelAnimationFrame(animationId);
Why rAF Over setInterval for Animation? requestAnimationFrame syncs with the display refresh rate, pauses when the tab is hidden (saving CPU/battery), and avoids tearing. setInterval(fn, 16) does not do any of these things.
