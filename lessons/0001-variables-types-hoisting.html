1. The Three Ways to Declare a Variable

JavaScript has three keywords for variable declarations. This is frequently tested in interviews.

KeywordScopeHoisted?Re-declarable?Re-assignable?
varFunctionYes (as undefined)YesYes
letBlockYes (TDZ — not accessible)NoYes
constBlockYes (TDZ — not accessible)NoNo *

* const prevents re-assignment, not mutation. You can still modify properties of a const object.

Rule of thumb Default to const. Use let only when you know the value will change. Never use var in new code — its function scope and hoisting behavior cause subtle bugs. 
const name = "Karan";
// name = "Someone else"; // ❌ TypeError: Assignment to constant variable

const user = { age: 22 };
user.age = 23; // ✅ Fine — we're mutating the object, not re-assigning the variable
user = {}; // ❌ TypeError — re-assignment is forbidden

Primitives vs. Objects: The Critical Difference

Primitives are copied by value. Objects are copied by reference.

// PRIMITIVES — copied by value
let a = 5;
let b = a;
b = 10;
console.log(a); // 5 — a is unchanged

// OBJECTS — copied by reference
let obj1 = { score: 5 };
let obj2 = obj1;
obj2.score = 10;
console.log(obj1.score); // 10 — both point to the SAME object
Interview Trap When you compare two objects with ===, JS compares their references, not their contents. {} === {} is false. Two separate empty objects are not the same object.

3. typeof — and Its Famous Bug

Use typeof to check a variable's type at runtime.

typeof "hello"     // "string"
typeof 42          // "number"
typeof true        // "boolean"
typeof undefined   // "undefined"
typeof Symbol()    // "symbol"
typeof {}          // "object"
typeof []          // "object" ← arrays are objects
typeof function(){} // "function" ← special case
typeof null        // "object" ← THE FAMOUS BUG
Known Language Bug typeof null === "object" is a bug from 1995 that can never be fixed without breaking the web. To check for null: use value === null, not typeof.

To check if something is an array: use Array.isArray(value).

4. Hoisting — What JavaScript Does Before Running Your Code

Before executing any code, the JavaScript engine hoists (moves to the top) declarations. The behavior differs for var, let/const, and functions.

var Hoisting

The declaration is hoisted and initialized to undefined. The assignment stays in place.

console.log(x); // undefined — NOT a ReferenceError
var x = 5;
console.log(x); // 5

// JS sees it as:
var x;           // declaration hoisted
console.log(x); // undefined
x = 5;           // assignment stays here

let and const — The Temporal Dead Zone (TDZ)

let and const are also hoisted, but they are not initialized. Accessing them before the declaration throws a ReferenceError. The time between the start of the scope and the declaration line is called the Temporal Dead Zone.

console.log(y); // ❌ ReferenceError: Cannot access 'y' before initialization
let y = 10;

// Same for const:
console.log(z); // ❌ ReferenceError
const z = 20;

Function Declaration Hoisting

Function declarations are fully hoisted — both the name and the body. You can call them before they appear in the code.

greet(); // ✅ "Hello!" — fully hoisted

function greet() {
  console.log("Hello!");
}

// But function EXPRESSIONS are NOT fully hoisted:
sayBye(); // ❌ TypeError: sayBye is not a function
var sayBye = function() { console.log("Bye!"); }
// Only 'var sayBye' is hoisted (as undefined), not the function itself
Mental Model Think of JavaScript execution in two phases:
1. Creation phase — scan for declarations, set them up in memory
2. Execution phase — run the code line by line
Hoisting is just what happens in phase 1.

5. Type Coercion — JavaScript's Surprising Conversions

JS will automatically convert types in certain operations. This is called implicit coercion and is a major source of bugs and interview questions.

// == uses coercion; === does not
0 == ""         // true  (both coerce to falsy)
0 === ""        // false (different types)
null == undefined  // true  (special rule)
null === undefined // false
"5" + 3        // "53"  (+ with string = concatenation)
"5" - 3        // 2     (- always converts to number)
true + true    // 2     (booleans coerce to numbers)
The Rule Always use === (strict equality) in production code. Use == only when you deliberately want type coercion — and always comment why.

Falsy and Truthy Values

In boolean context, every value is either truthy or falsy. Only 6 values are falsy:

// The 6 falsy values:
false
0          // also -0 and 0n (BigInt zero)
""         // empty string
null
undefined
NaN

// Everything else is truthy — including:
"0"        // ← truthy! Non-empty string
[]         // ← truthy! Empty array
{}         // ← truthy! Empty object
